# PPT HTML Deck Generation Prompt

아래 파일은 다른 LLM에 그대로 넣어서, `reveal.js` 기반의 12장짜리 standalone HTML 발표자료를 생성하기 위한 초장문 프롬프트입니다.

## Slide-to-Library Lookup

| 슬라이드 | 역할 | 메인 렌더링 라이브러리 | 보조 |
|---|---|---|---|
| 1 | 오프닝/문제 정의 | `reveal.js` | `GSAP` |
| 2 | MCP 직관 설명 | `Mermaid` | `GSAP` |
| 3 | 양자화학이 왜 필요한가 | `3Dmol.js` | `GSAP` |
| 4 | 기존 실험자 병목 | `ECharts` | `Rough Notation`, `GSAP` |
| 5 | 인터페이스 전환 필요성 | `Mermaid` | `reveal.js Auto-Animate` |
| 6 | 프로젝트 핵심 아이디어 | `Mermaid` | `GSAP` |
| 7 | 사용자 여정 | `Mermaid User Journey` | `3Dmol.js` |
| 8 | 시스템 내부 흐름 | `Mermaid` | `GSAP` |
| 9 | 결과 화면 가치 | `3Dmol.js` | `GSAP` |
| 10 | 실제 가치 | `ECharts` | `GSAP` |
| 11 | 한계와 확장 | `ECharts` 또는 `Mermaid` | `GSAP` |
| 12 | 결론 | `reveal.js Auto-Animate` | `GSAP`, `Rough Notation` |

## Prompt

```text
너는 프레젠테이션 스토리텔링, 정보설계, 프론트엔드 구현, HTML 기반 발표자료 제작에 강한 시니어 프레젠테이션 엔지니어다.

지금부터 해야 할 일:
단일 파일 `index.html` 형태의 standalone 발표자료를 만들어라.
발표자료는 reveal.js 기반의 16:9 HTML 슬라이드 덱이어야 하며,
각 슬라이드에 맞는 최신 JavaScript 렌더링 라이브러리를 적절히 삽입하여,
정적인 슬라이드가 아니라 “설명력이 높은 인터랙티브 발표자료”로 완성하라.

매우 중요:
- 이 발표는 기술스택 자랑 발표가 아니다.
- 핵심 스토리라인은 반드시 다음 순서를 따른다:
  MCP → 양자화학 → 기존 실험자들의 병목(DFT를 직접 다루기 어려움) → 왜 새로운 인터페이스가 필요한가 → 이 프로젝트가 무엇을 하는가 → 사용자 흐름 → 시스템 흐름 → 결과 가치 → 한계와 확장 → 결론
- 기술 용어는 최소화하되, 의미는 정확해야 한다.
- 청중은 양자화학 전문가만이 아니라 연구자, 실험자, 개발자, 의사결정자가 섞여 있다고 가정한다.
- 발표자의 설명 톤은 “과외 선생님처럼 차근차근 설명하는 톤”이어야 한다.
- 과장된 스타트업 피치처럼 보이면 안 되고, 연구 현장의 병목을 잘 이해하고 설계한 프로젝트처럼 보여야 한다.
- 전체 발표는 “단순 웹앱”이 아니라 “실험자가 계산의 문턱을 넘도록 돕는 자연어 기반 양자화학 작업대”라는 관점으로 설계하라.

프로젝트 핵심 맥락:
- 사용자는 자연어로 분자 계산을 요청할 수 있다.
- 시스템은 요청 의도를 해석해 계산 종류를 정한다.
- 분자명을 실제 계산 가능한 구조로 변환한다.
- 양자화학 계산을 수행한다.
- 결과를 summary, geometry, charges, orbital, ESP, history, progress 등의 형태로 정리해 보여준다.
- 웹 UI에서 채팅, 작업 이력, 진행 상태, 3D 시각화를 제공한다.
- 목표는 계산 전문가가 아닌 실험자도 기본적인 계산 워크플로를 쉽게 활용하도록 만드는 것이다.

최종 산출물 요구사항:
- 출력은 완전한 `index.html` 코드 하나만 내놓아라.
- 별도의 빌드 도구 없이 브라우저에서 바로 열 수 있어야 한다.
- CDN 기반 라이브러리 로딩을 사용하라.
- reveal.js를 기반으로 슬라이드를 구성하라.
- JS 렌더링 라이브러리는 다음 조합을 기본으로 사용하라:
  - reveal.js: 발표 셸
  - GSAP: 모션
  - Mermaid: 흐름도/블록도/User Journey
  - Apache ECharts: 병목/가치/확장 차트
  - 3Dmol.js: 분자 시각화
  - Rough Notation: 핵심 문장 강조
- React, Vue, Svelte 같은 프레임워크는 사용하지 마라.
- D3는 정말 필요한 경우에만 쓰고, 기본적으로 쓰지 마라.
- Three.js는 쓰지 마라. 분자 관련 3D는 3Dmol.js로 해결하라.
- Motion은 React 기반이 아니므로 이번 구현에서는 사용하지 마라.
- 한 슬라이드에 보조 라이브러리는 최대 2개까지만 겹쳐라.
- 전역 공통 라이브러리는 reveal.js, GSAP만 상시 활성화하고,
  Mermaid, ECharts, 3Dmol.js, Rough Notation은 슬라이드 진입 시 lazy-init 하라.

출력 코드 구조 요구사항:
- HTML 문서 전체를 완성하라.
- `<head>`에는 메타태그, 폰트, CSS 변수, 테마 스타일을 넣어라.
- `<body>`에는 reveal.js 컨테이너와 12개 `<section>` 슬라이드를 넣어라.
- 하단 `<script>`에서 각 라이브러리 로드 이후 초기화 로직을 작성하라.
- `Reveal.on('slidechanged', ...)` 기반으로 슬라이드별 애니메이션과 렌더링 초기화를 연결하라.
- `Reveal.on('ready', ...)`에서도 첫 슬라이드 렌더를 처리하라.
- ECharts와 3Dmol.js는 resize 대응을 하라.
- 동일 슬라이드 재진입 시 중복 초기화하지 않도록 가드 로직을 넣어라.
- Mermaid는 한 번만 초기화하고, 필요한 슬라이드에서만 렌더하라.
- 네트워크 실패 시 슬라이드가 완전히 깨지지 않도록 fallback 메시지를 넣어라.
- 콘솔 로그를 과도하게 남기지 마라.
- HTML 코드 안에 충분한 주석을 넣어, 왜 그런 구조로 만들었는지 설명하라.

디자인 방향:
- 기본 무드는 “연구실 작업대 + 과학적 명료함 + 따뜻한 설명력”이다.
- 너무 차갑고 기업적인 파랑/보라 위주의 테마는 피하라.
- 권장 팔레트:
  - 배경: warm ivory / paper tone
  - 주 텍스트: carbon ink
  - 포인트 1: oxidized teal
  - 포인트 2: rust / amber
  - 경고/병목: muted crimson
- CSS 변수로 색 체계를 정의하라.
- 폰트는 한국어 가독성을 우선하라.
- 권장 조합:
  - 제목: Space Grotesk 또는 IBM Plex Sans
  - 본문: Noto Sans KR
  - 보조/모노: IBM Plex Mono 또는 JetBrains Mono
- 배경은 단색 흰색으로 끝내지 말고, 아주 얕은 그리드, 종이 질감, 방사형 그라디언트 등을 사용해 깊이를 줘라.
- 그러나 과하게 화려하면 안 된다.
- 발표용으로 멀리서도 읽혀야 하므로 제목은 크고 짧게, 문장은 적되 의미는 강하게 하라.
- 한 슬라이드에 “핵심 시각 1개”를 원칙으로 하라.
- 정보가 많은 슬라이드는 좌우 분할, 상하 분할을 명확히 하라.
- reveal.js의 `fragments`는 필요한 곳에만 쓰고 남용하지 마라.
- `data-auto-animate`는 5번, 12번 슬라이드에 적극 활용하라.

발표자 노트:
- 각 슬라이드에 가능하면 `aside.notes` 또는 데이터 속성으로 짧은 발표자 힌트를 넣어라.
- 발표자 노트는 한국어로 작성하라.
- 노트는 문장형 2~4줄 정도면 충분하다.

라이브러리 룩업테이블:
- 슬라이드 1: reveal.js + GSAP
- 슬라이드 2: Mermaid + GSAP
- 슬라이드 3: 3Dmol.js + GSAP
- 슬라이드 4: ECharts + Rough Notation + GSAP
- 슬라이드 5: Mermaid + reveal.js Auto-Animate
- 슬라이드 6: Mermaid + GSAP
- 슬라이드 7: Mermaid User Journey + 3Dmol.js
- 슬라이드 8: Mermaid + GSAP
- 슬라이드 9: 3Dmol.js + GSAP
- 슬라이드 10: ECharts + GSAP
- 슬라이드 11: ECharts 또는 Mermaid + GSAP
- 슬라이드 12: reveal.js Auto-Animate + GSAP + Rough Notation

중요한 구현 원칙:
- 슬라이드는 총 12장만 만들어라.
- 수직 슬라이드는 만들지 마라.
- reveal.js 테마는 기본 테마를 그대로 쓰지 말고 커스텀 CSS로 재정의하라.
- `fragment`는 bullet reveal이 아니라 논리 단계 reveal에만 사용하라.
- 모든 슬라이드는 한국어 텍스트로 작성하라.
- 영어는 보조적으로만 사용하라.
- 코드 블록은 거의 넣지 마라.
- 이 발표는 기술 데모보다 “문제 해결 스토리”가 우선이다.
- 슬라이드 내 텍스트는 너무 길지 않되, 시각자료와 조합했을 때 메시지가 충분히 전달되도록 하라.
- 발표자가 말로 풀어줄 여지를 남기되, 슬라이드만 봐도 맥락은 이해되게 하라.

이제 아래의 12개 슬라이드를 정확히 구현하라.

슬라이드 1. 문제를 한 줄로 정의하는 오프닝
목적:
- 청중이 첫 화면에서 이 프로젝트의 존재 이유를 즉시 이해하게 만든다.

핵심 메시지:
- 이 프로젝트는 실험자가 양자화학 계산의 문턱을 넘도록 돕는 자연어 기반 작업대다.

반드시 포함할 내용:
- 강한 제목 1개
- 짧은 부제 1개
- “사용자의 질문이 계산과 시각화로 이어진다”는 한 줄 메시지
- 아주 짧은 예시 문장 1개
  예: “물의 HOMO 보여줘”

권장 레이아웃:
- 좌측 60%: 제목, 부제, 핵심 메시지
- 우측 40%: 자연어 입력 → 계산 → 3D 결과의 미니 흐름
- 큰 타이포 중심
- 배경은 얕은 연구 노트/그리드 느낌

렌더링:
- reveal.js 기본 슬라이드
- GSAP으로 제목, 부제, 우측 흐름 요소를 순차적으로 등장시켜라
- Rough Notation은 여기서는 쓰지 마라

HTML 요소 id 예시:
- `#heroTitle`
- `#heroSub`
- `#heroFlow`
- `#heroPrompt`

슬라이드 2. MCP를 기술이 아니라 연결 계층으로 설명
목적:
- MCP를 어렵게 정의하지 않고 왜 필요한지 직관적으로 이해시킨다.

핵심 메시지:
- MCP는 사람의 자연어와 계산 도구 사이를 연결하는 중간 계층이다.

반드시 포함할 내용:
- “사람의 의도”
- “중간 연결 계층”
- “계산 도구/시각화 도구”
- 기존 방식과 대비되는 한 줄 설명
- ‘도구를 직접 조작하는 대신, 의도를 전달하고 시스템이 연결한다’는 관점

권장 레이아웃:
- 중앙 정렬형 3단 블록 다이어그램
- 상단에 간단한 제목
- 하단에 핵심 해석 문장 1개

렌더링:
- Mermaid block diagram 또는 flowchart 사용
- GSAP으로 다이어그램 전체가 살짝 상승하며 fade-in
- 텍스트는 최소화

HTML 요소 id 예시:
- `#mcpDiagram`
- `#mcpCaption`

슬라이드 3. 왜 실험자에게 양자화학이 필요한가
목적:
- 양자화학이 일부 전문가 전용이 아니라 실험자의 질문에 답하는 도구임을 보여준다.

핵심 메시지:
- 실험자는 계산 이론 자체보다, 분자의 성질과 반응성을 이해하기 위해 계산이 필요하다.

반드시 포함할 내용:
- 실험자의 질문 4~5개:
  - 어디가 반응하기 쉬운가
  - 전자 밀도는 어디에 몰려 있는가
  - HOMO/LUMO 관점에서 해석 가능한가
  - ESP로 표면 특성을 볼 수 있는가
  - 구조가 안정한가
- 분자 3D 시각화 1개
- “질문”과 “계산이 줄 수 있는 답”의 대응 관계

권장 레이아웃:
- 좌측 45%: 실험자의 질문 카드
- 우측 55%: 3Dmol.js 분자 뷰어
- 오른쪽 아래에 “전자구조 / ESP / 전하 / 안정성” 키워드 배치

렌더링:
- 3Dmol.js로 간단한 분자 구조를 회전시켜라
- 수소/탄소/산소가 보이는 단순 분자면 충분하다
- GSAP으로 질문 카드가 하나씩 올라오게 하라

HTML 요소 id 예시:
- `#chemQuestions`
- `#molNeedViewer`
- `#chemKeywords`

슬라이드 4. 기존 실험자들의 병목
목적:
- 이 발표의 가장 중요한 공감 슬라이드.
- 왜 이런 프로젝트가 필요한지 감정적으로 납득시킨다.

핵심 메시지:
- 병목은 DFT 그 자체보다, 구조 준비부터 해석과 시각화까지 이어지는 전체 워크플로의 마찰에 있다.

반드시 포함할 내용:
- 병목 단계 6개 이상:
  - 구조 파일 준비
  - 입력 작성
  - method/basis 선택
  - 전하/다중도 판단
  - 계산 실행과 상태 확인
  - 출력 해석
  - 오비탈/ESP 시각화
- “실험자는 화학을 모르기 때문이 아니라, 계산 워크플로 전체를 혼자 감당해야 해서 막힌다”는 메시지
- 병목을 강조하는 시각적 표시

권장 레이아웃:
- 좌측: 워크플로/샌키/단계 플로우 시각화
- 우측: 병목 설명 3줄
- 병목 키워드 몇 개에 손그림 강조

렌더링:
- ECharts로 흐름형 시각화 또는 단계별 막대 시각화 구현
- Rough Notation으로 핵심 병목 문장을 강조
- GSAP으로 차트가 순차 등장

HTML 요소 id 예시:
- `#bottleneckChart`
- `#bottleneckQuote`
- `#bottleneckNotes`

슬라이드 5. 그래서 인터페이스가 바뀌어야 한다
목적:
- 기존 방식과 새로운 방식의 차이를 명확히 보여준다.

핵심 메시지:
- 필요한 것은 더 복잡한 계산기가 아니라, 의도를 중심으로 동작하는 자연어 기반 인터페이스다.

추가 스타일/구현 지시:
- 슬라이드 번호는 너무 튀지 않게 우하단에 작게 넣어라.
- 진행 표시(progress bar)는 reveal.js 기본 또는 커스텀으로 제공하라.
- 전체 발표 길이는 7~10분 발표용 템포에 맞게 구성하라.
- 각 슬라이드에 등장 애니메이션은 1.5초 이내로 절제하라.
- 모든 애니메이션은 발표 메시지를 돕는 수준이어야 하며 장난스럽지 않아야 한다.
- 키보드 조작과 모바일 너비에서도 최소한 깨지지 않게 하라.
- 하지만 1순위는 데스크탑 발표 화면이다.
- CSS 클래스명은 의미 있게 지어라.
- JavaScript는 기능별 함수로 분리하라.
- 함수 예시:
  - `initMermaidSlide()`
  - `initBottleneckChart()`
  - `initMoleculeViewer()`
  - `initValueChart()`
  - `runSlideAnimation(index)`
- 슬라이드별 초기화 상태를 저장하는 맵을 두어 재실행을 방지하라.
- 라이브러리 로드 실패 시 각 영역에 “시각화를 불러오지 못했습니다” 같은 fallback 텍스트를 출력하라.
```
