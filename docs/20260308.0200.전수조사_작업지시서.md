# QCViz-MCP Enterprise v2 전수조사 기반 프론트엔드/백엔드 대개편 작업 지시서

## 1. 현재 시스템의 구조적 문제점 전수조사 요약 (Pain Points)

### 1-1. 프론트엔드 (UI/UX & JS 아키텍처)
*   **상태 관리(State Management)의 파편화**: `viewer.js`, `results.js`, `chat.js`가 각각 독립적인 state를 가지고 있어, Job이 완료되거나 이력을 전환할 때 상태 동기화가 깨지거나 불완전하게 복원됨. (예: Job History 클릭 시 뷰어의 컨트롤 바 값과 실제 렌더링 상태가 불일치).
*   **시각적 노이즈 및 공간 낭비**: `index.html`의 뷰어 상단에 자리 잡은 `hero-info`와 중복된 설명 텍스트, 장황한 메트릭 등이 실제 전문가(연구원)의 작업 시야를 방해함. UI가 "데모" 수준에 머물러 있음.
*   **인터랙션 버그 및 예외 처리 미흡**:
    *   원자(Atom) 및 전하(Charge) 레이블 토글 시 3Dmol.js 내부 객체 참조가 꼬여 잔상이 남거나 지워지지 않는 버그 존재.
    *   오비탈 렌더링 후 ESP를 연속 렌더링할 때 이전 볼륨 데이터가 메모리에 남아 뷰어가 무거워지는 현상.
    *   스타일(Ball+Stick, Sphere 등) 변경 시 현재 적용된 Isosurface 설정이 초기화되는 문제.
*   **Job History(계산 이력) UX 부재**: 단순히 `results.js`의 표(Table) 형태로만 제공되며, 사용자가 과거 계산 결과의 3D 형태(오비탈, ESP)를 "클릭 한 번으로 100% 복원"하는 기능이 불완전함.

### 1-2. 백엔드 (API & Job Manager)
*   **Job Manager의 메모리/데이터 관리**: `InMemoryJobManager`가 과거 결과의 대용량 Base64 큐브 데이터를 모두 들고 있어 누수(Memory Leak) 우려가 있음. LRU 캐시나 파일 시스템 기반의 오프로딩(Offloading) 전략 부재.
*   **데이터 정규화 미세 오류**: 프론트엔드의 "Charge Labels" 기능을 100% 지원하려면, 각 원자의 3D 좌표(XYZ)와 Mulliken Charge 값이 완벽히 매핑된 JSON 배열이 필요한데, 현재 컨트랙트가 다소 느슨함.
*   **Fallback 로직의 한계**: 자연어 파싱(`pyscf_runner.py`의 `_fallback_extract_structure_query`)이 많이 개선되었으나, 여전히 복잡한 영어/한국어 혼용 지시(예: "물분자의 HOMO-1 오비탈을 투명도 0.5로 그려줘")에서 인텐트(Intent)와 파라미터를 완벽히 분리하지 못하는 케이스 존재.

## 2. 통합 대개편 작업 지시 (Prompt for LLM)

당신은 실리콘밸리 탑티어 풀스택 아키텍트이자 UI/UX 마스터입니다. 제공된 QCViz-MCP의 프론트엔드/백엔드 코드를 기반으로, 시스템을 **'전문가용 엔터프라이즈 SaaS'** 수준으로 완전히 탈바꿈시켜야 합니다. 아래의 전수조사된 문제점들을 100% 해결하는 **완벽한 Full Code**를 작성해주세요.

### 🎯 [작업 목표 1: 프론트엔드 아키텍처 및 상태 동기화]
1. `chat.js`, `results.js`, `viewer.js` 간의 파편화된 상태를 통합 관리할 수 있는 **중앙 집중형 이벤트 버스(또는 단일 State 객체 공유 방식)**를 구현하세요.
2. 특정 Job History 항목을 클릭했을 때, 해당 Job의 결과 데이터(JSON), 컨트롤 바(Slider)의 수치, 3Dmol.js의 렌더링 상태(오비탈/ESP)가 **한 치의 오차도 없이 과거 상태 그대로 즉각 복원**되도록 로직을 전면 재작성하세요.
3. 메모리 누수 방지: 3Dmol.js에서 오비탈/ESP 볼륨 데이터를 전환할 때 기존 Shape와 Surface 객체를 깔끔하게 소멸(Dispose)시키는 로직을 추가하세요.

### 🎯 [작업 목표 2: UI 다이어트 및 엔터프라이즈 미학 적용]
1. **극강의 미니멀리즘**: `index.html` 뷰어 상단의 `hero-info`(기능 요약) 및 불필요한 설명 텍스트를 완전히 삭제하세요. 연구원이 오직 3D 뷰어와 데이터에만 집중할 수 있도록 공간을 확보하세요.
2. **Glassmorphism & 세련된 그리드**: `style.css`를 대폭 수정하여, 다크/라이트 톤이 고급스럽게 조화된 Apple/Vercel 스타일의 엔터프라이즈 디자인을 적용하세요. 툴바, 슬라이더, 토글 버튼이 매우 직관적이고 부드러운 애니메이션을 갖도록 만드세요.
3. **직관적인 컨트롤**: 원자/전하 레이블 토글, 스크린샷, 스타일 변경 버튼이 버그 없이 100% 정확하게 3D 뷰어와 연동되도록 `viewer.js`의 DOM 바인딩을 고도화하세요.

### 🎯 [작업 목표 3: 백엔드 최적화 및 페이로드 정밀화]
1. `pyscf_runner.py`의 `_populate_scf_fields`를 수정하여, 프론트엔드에서 원자별 전하 레이블(Charge Labels)을 3D 좌표 위에 정확히 띄울 수 있도록 `atoms` 배열 내에 `charge` 필드를 병합(Merge)하여 내려주는 로직을 추가하세요.
2. `InMemoryJobManager`가 대용량 큐브 데이터(Base64)를 처리할 때 시스템이 뻗지 않도록, 필요 시 LRU 메커니즘이나 페이로드 최적화 로직을 반영하세요.
3. LLM Planner가 실패했을 때 작동하는 정규식(Regex) Fallback을 극한으로 고도화하여, `"벤젠의 HOMO-1 오비탈 그려줘"` 같은 복잡한 한국어 지시문에서도 정확한 파라미터를 뽑아내도록 개선하세요.
