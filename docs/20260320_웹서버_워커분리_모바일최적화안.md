# QCViz v3 다음 단계 고도화안

## 1. 왜 지금 구조가 버거운가

현재 서비스는 FastAPI 웹서버와 PySCF 계산이 같은 머신, 같은 앱 프로세스 계열 안에서 함께 돌아가고 있습니다. 이 구조는 구현이 단순하고 초기 개발에는 빠르지만, 사용자가 여러 명 붙거나 무거운 계산이 겹치면 웹 응답성과 계산 안정성이 동시에 흔들립니다. 특히 PySCF와 NumPy/BLAS 계열은 내부적으로도 CPU 스레드를 쓰기 때문에, 단순히 Python 레벨 worker 수만 늘리면 실제로는 CPU oversubscription이 발생해서 사이트가 멈춘 것처럼 느껴질 수 있습니다.

## 2. 목표 아키텍처

다음 단계의 목표는 `웹 응답 계층`과 `계산 실행 계층`을 분리하는 것입니다. 웹서버는 요청 해석, 세션 상태, queue 표시, 결과 조회만 맡고, 실제 계산은 외부 작업 큐를 통해 별도 worker 프로세스가 수행해야 합니다. 이때 메시지 브로커와 작업 큐는 Redis + arq 조합이 현재 코드베이스와 가장 자연스럽게 맞습니다. FastAPI는 비동기 웹에 이미 잘 맞고 있고, arq는 Python async/Redis 기반이라 지금의 `InMemoryJobManager`를 단계적으로 치환하기 쉽습니다.

## 3. 권장 2단계 마이그레이션

1단계는 `단일 머신 분리`입니다. 같은 컴퓨터를 쓰더라도 웹 프로세스와 계산 워커를 분리해서 실행합니다. 웹은 `uvicorn`으로 가볍게 유지하고, worker는 Redis queue를 polling 하며 계산만 수행합니다. 이 단계에서는 DB 없이도 Redis + 파일 캐시만으로 충분히 시작할 수 있습니다. 2단계는 `다중 머신 분리`입니다. 웹서버는 공개 접속용 머신에 두고, 계산 워커는 별도 Linux 계산 노드로 옮깁니다. 이때 Redis는 공용 브로커 역할을 하며, 결과 artifact는 공용 스토리지 또는 DB/파일 시스템으로 연결합니다.

## 4. 구성 권장안

- Web/API: FastAPI + Uvicorn
- Queue/Broker: Redis
- Worker: arq
- Persistent metadata: PostgreSQL 또는 현재 단계에서는 JSON/SQLite 임시 유지 가능
- Result artifact: 파일 캐시 또는 object storage
- Monitoring: health endpoint + queue depth + worker heartbeat + per-job runtime metrics

이 조합의 장점은 지금 코드와의 거리감이 작다는 점입니다. `InMemoryJobManager.submit()`과 `wait()` 패턴을 `Redis enqueue / poll result` 형태로 바꾸면 라우트 계약을 거의 유지할 수 있습니다.

## 5. 백엔드 분리 방식

현재 `compute.py` 안의 `InMemoryJobManager`는 웹 프로세스 내부에서 `ThreadPoolExecutor`를 사용합니다. 이를 다음처럼 바꾸는 것이 좋습니다.

- 웹서버:
  - 요청 수신
  - planner 실행
  - payload validation
  - queue enqueue
  - job metadata 저장
  - status/result/events 조회
- 계산 워커:
  - Redis에서 job pickup
  - structure resolve / PySCF / advisor 실행
  - progress/event/result 저장
  - 실패 시 retry policy 또는 terminal failure 기록

즉 웹은 orchestration, worker는 execution으로 역할을 분리합니다.

## 6. job 상태 저장 구조

현재도 `queued / running / completed / failed` 구조와 event log가 이미 존재하므로, 이를 유지한 채 저장 위치만 바꾸면 됩니다. 최소 권장 필드는 다음과 같습니다.

- `job_id`
- `status`
- `submitted_at`
- `started_at`
- `ended_at`
- `payload`
- `queue_name`
- `worker_id`
- `resource_profile`
- `events`
- `result_summary`
- `result_path`
- `error`

이 구조를 PostgreSQL JSONB로 저장하면, 나중에 “최근 실패 job”, “평균 계산 시간”, “분자별 반복 요청” 같은 운영 분석도 쉬워집니다.

## 7. Redis/arq를 쓰는 이유

arq는 지금 코드베이스에 비해 도입 비용이 낮습니다. Celery보다 가볍고, FastAPI/async 조합과 잘 맞고, Redis만 있으면 바로 queue depth와 worker 수를 모니터링할 수 있습니다. 특히 지금 필요한 것은 복잡한 DAG보다 `무거운 계산 job을 안전하게 순차/제한 실행`하는 것이므로, arq가 충분합니다. 이후 human-in-the-loop나 장시간 durable workflow가 정말 필요해지면 Temporal/LangGraph를 검토해도 늦지 않습니다.

## 8. 자원 제한 전략

job별 자원 제한은 꼭 들어가야 합니다. 단순히 worker 수만 줄이는 것으로는 부족합니다. 최소한 다음 레벨의 제한이 필요합니다.

- worker process 수 제한
- job 유형별 동시 실행 제한
- BLAS/OpenMP thread 제한
- job timeout
- memory guard
- queue priority

예를 들어 geometry optimization은 single-point보다 훨씬 무겁기 때문에, `geometry_optimization`은 동시 1개, `single_point`는 동시 2개 같은 정책이 필요할 수 있습니다.

## 9. 추천 resource profile

- `single_point`: CPU 1, medium memory, timeout medium
- `orbital_preview`: CPU 1, medium memory, timeout medium
- `esp_map`: CPU 1, medium-high memory, timeout medium-high
- `geometry_optimization`: CPU 1, high memory, timeout high
- `analyze`: 내부적으로 여러 결과를 묶으므로 high profile

지금 단계에서는 worker 수를 늘리는 것보다, 각 worker가 내부 BLAS thread를 1로 고정하는 것이 실제 응답성에 더 유리합니다.

## 10. queue UX 고도화 방향

이번에 넣은 기본 대기열 표시보다 한 단계 더 가려면 아래 정보를 명시적으로 보여주는 것이 좋습니다.

- 내 앞에 몇 개 있는지
- 현재 몇 개가 실행 중인지
- 평균 대기 시간
- 예상 시작 시각
- 현재 내 job의 priority
- 취소 가능 여부

이 정보는 Assistant 채팅 카드, History 카드, 별도 queue drawer에서 동시에 재사용할 수 있도록 하나의 queue summary schema로 정리하는 것이 좋습니다.

## 11. 사용자 체감 UX

사용자 입장에서 중요한 것은 “왜 안 움직이는지 모르는 상태”를 없애는 것입니다. 그래서 queue overlay는 단순 숫자보다 메시지형이 좋습니다. 예를 들면:

- `현재 다른 계산 1개 실행 중입니다`
- `내 앞 대기열 2개`
- `예상 대기 3~5분`
- `구조는 먼저 확인 가능합니다`

즉 시스템이 바쁘더라도 사용자는 진행 상황을 이해할 수 있어야 합니다.

## 12. 구조 미리보기와 계산 본체 분리

이미 일부 반영했듯이, 구조 해석이 끝나면 분자 구조부터 먼저 보여주는 것은 매우 중요합니다. 다음 단계에서는 이걸 더 분리해서 `resolve preview job`과 `heavy compute job`을 별도 단계로 나누는 것이 좋습니다. 그러면 사용자 입장에서는 먼저 구조와 계산 계획을 확인한 뒤, 본 계산은 queue에 넣는 방식으로 UX가 더 부드러워집니다.

## 13. 모바일 UX와 queue UX의 연결

모바일에서는 대기열, 상태, 진행 상황이 더 중요합니다. 화면이 좁기 때문에 사용자는 한 번에 전체 대시보드를 보지 못합니다. 따라서 모바일에서는 다음 3가지가 핵심입니다.

- sticky panel navigation
- compact queue summary
- 결과 테이블의 가로 스크롤 안전성

즉 데스크탑의 “모든 패널 동시 노출”보다, 모바일의 “현재 맥락을 잃지 않도록 이어주는 상태 표시”가 더 중요합니다.

## 14. 구현 순서 제안

### Phase A: 같은 머신에서 웹/워커 분리

- Redis 도입
- `InMemoryJobManager` 인터페이스 유지
- 내부 구현만 Redis enqueue 방식으로 교체
- worker 프로세스 별도 실행
- 결과/이벤트 저장소 추상화

### Phase B: job resource profile 도입

- job_type별 timeout
- job_type별 concurrency cap
- worker별 BLAS thread=1 강제
- queue priority 도입

### Phase C: 운영 가시성 강화

- `/compute/health`에 queue depth, worker heartbeat, stale job count
- `/chat/health`에 ws/queue summary
- average wait time / average run time

### Phase D: 멀티 머신 분리

- 웹 서버와 worker host 분리
- Redis 공용화
- 결과 artifact 저장소 분리

## 16. 최종 판단

지금 구조에서 가장 먼저 중요한 것은 worker 수를 무작정 늘리는 것이 아니라, `웹 응답성 보호`와 `예측 가능한 queue 운영`입니다. 그래서 다음 단계는 분명히 `웹/계산 분리 + Redis/arq + 자원 제한 + queue UX 고도화` 방향으로 가는 것이 맞습니다. 이 경로는 현재 코드와도 잘 이어지고, 연구실 내부 실사용 단계로 넘어갈 때 가장 현실적인 확장 경로입니다.
