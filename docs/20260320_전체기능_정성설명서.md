# QCViz-MCP v3 전체 기능 정성 설명서

기준 시점: 2026-03-20  
기준 범위: `src/qcviz_mcp` 전체 코드, `web`, `llm`, `services`, `compute`, `advisor`, `tools`, `backends`, `analysis`, `renderers`, `execution`, `tests` 계층을 포함한 현재 저장소 기준  
문서 목적: 이 프로젝트에 이미 들어와 있는 기능을 단순 목록이 아니라, 실제 사용 경험과 설계 의도를 함께 읽을 수 있도록 기능별 한 문단 설명으로 풀어내는 것

## 1. 프로젝트 전체 성격

### 1.1 자연어 기반 양자화학 워크스페이스
이 프로젝트의 가장 큰 기능은 사용자가 분자명이나 계산 키워드를 자연어로 입력하면, 그 요청을 구조 해석, 계산 준비, 양자화학 실행, 시각화, 결과 설명까지 이어지는 하나의 워크플로우로 바꿔 준다는 점이다. 단순히 계산만 던져주는 서버가 아니라, 사용자의 거친 요청을 실제 실행 가능한 작업으로 다듬고, 필요한 경우 중간 질문을 던지고, 최종적으로 사람이 이해할 수 있는 방식으로 다시 보여 주는 작업공간 성격이 강하다. 그래서 이 프로젝트는 “PySCF 웹 래퍼”보다는 “LLM과 화학 백엔드를 결합한 대화형 계산 환경”에 더 가깝다.

### 1.2 웹앱과 MCP 서버의 이중 인터페이스
이 저장소는 브라우저에서 직접 쓰는 웹 애플리케이션과, 외부 에이전트나 툴 체인에서 붙일 수 있는 MCP 서버를 함께 제공한다. 즉 사람 사용자는 웹 UI에서 계산을 요청할 수 있고, AI 에이전트나 자동화 환경은 등록된 MCP 툴을 통해 같은 계산 기능에 접근할 수 있다. 이중 인터페이스 구조 덕분에 제품은 “사람용 UI”와 “에이전트용 실행 인터페이스”를 동시에 갖게 되며, 향후 자동화 실험 파이프라인이나 연구 보조 에이전트와도 자연스럽게 결합될 수 있다.

## 2. 웹 애플리케이션 진입과 공용 라우팅 기능

### 2.1 FastAPI 웹앱 생성 기능
`web/app.py`는 이 프로젝트의 HTTP 진입점이며, FastAPI 애플리케이션을 조립하고 라우터, 정적 파일, 템플릿, CORS, 헬스체크, API 별칭 경로를 한 번에 묶어낸다. 이 기능의 좋은 점은 실제 서비스 환경에서 필요한 기본 골격이 이미 다 준비되어 있다는 것이다. 개발자가 단지 계산 함수만 넣는 수준이 아니라, 배포 가능한 웹앱 형태로 시작할 수 있고, `/`, `/api`, `/health`, `/static`, `/api/static` 같은 표면이 정리된 상태로 제공된다.

### 2.2 API 별칭 경로 제공 기능
이 프로젝트는 `/chat`, `/compute` 같은 기본 경로뿐 아니라 `/api/chat`, `/api/compute` 같은 별칭도 같이 제공한다. 이 기능은 생각보다 실용적인데, 프론트엔드와 외부 자동화 스크립트가 서로 다른 URL 습관을 가지고 있어도 같은 기능에 접근할 수 있게 해주기 때문이다. 운영 중 버전 전환이나 프록시 설정이 바뀌어도 사용자 코드와 UI가 덜 깨지고, 테스트 코드 역시 서로 다른 엔드포인트 표현을 한 번에 검증할 수 있다.

### 2.3 루트 인덱스 및 템플릿 폴백 기능
`/` 경로는 정식 `index.html` 템플릿을 우선 보여 주고, 템플릿이 없을 경우에도 최소한의 안내 HTML을 폴백으로 제공한다. 이 기능은 서비스가 완전히 깨지지 않게 하는 장치로서 의미가 크다. 배포 실수로 템플릿이 누락되더라도 핵심 API와 WebSocket 엔드포인트가 살아 있다는 사실을 화면에서 곧바로 확인할 수 있어, 장애 상황에서도 문제를 빠르게 진단할 수 있다.

### 2.4 전역 헬스체크 기능
`/health`와 `/api/health`는 서버가 단순히 살아 있는지만 보는 엔드포인트가 아니라, 정적 파일과 템플릿 경로, 라우트 표면, 타임스탬프 같은 환경 정보를 같이 노출하는 기본 진단 기능이다. 이 기능 덕분에 운영자는 배포 직후 웹 자산이 제대로 마운트됐는지, 경로 테이블이 기대한 형태인지, 앱 버전이 무엇인지 빠르게 확인할 수 있다. 단순 ping 용도가 아니라 실제 배포 검증용 기능으로 쓸 수 있다.

## 3. 채팅 인터페이스 기능

### 3.1 REST 채팅 요청 처리 기능
`POST /chat`와 `POST /api/chat`는 사용자가 자연어로 계산 요청이나 설명 요청을 보냈을 때 가장 먼저 닿는 엔드포인트다. 이 기능은 “그냥 대답하는 챗봇”이 아니라, 요청을 분석해서 계산 잡을 큐에 넣을지, 바로 결과를 기다릴지, clarification이 필요한지, 설명형 응답으로 끝낼지를 결정한다. 즉 채팅 API는 인간 친화적인 입구 역할을 하면서도 내부적으로는 매우 구조화된 오케스트레이션 관문으로 작동한다.

### 3.2 WebSocket 실시간 채팅 기능
`/ws/chat`과 `/api/ws/chat`는 실시간 양방향 채팅을 지원해, 계산이 즉시 끝나지 않는 경우에도 진행 상태와 시스템 메시지를 끊기지 않고 전달한다. 이 기능은 특히 계산형 요청에서 UX 차이를 크게 만든다. 사용자는 단순히 로딩만 보는 것이 아니라, 잡 제출, 진행률, 중간 이벤트, 최종 결과까지 대화 흐름 안에서 이어서 받을 수 있고, 서버 입장에서도 장시간 작업을 좀 더 자연스럽게 다룰 수 있다.

### 3.3 세션 기반 멀티턴 대화 기능
채팅 라우트는 `session_id`를 기준으로 이전 plan, 이미 물어본 질문, 채운 슬롯값을 기억하는 세션 상태를 유지한다. 이 기능이 중요한 이유는 사용자가 한 번에 완벽한 명령을 입력하지 않아도 되기 때문이다. 예를 들어 “HOMO 보여줘”라고 말한 뒤 “물”이라고만 답해도, 시스템은 앞 문맥과 새 답변을 합쳐 최종 계산 요청으로 병합할 수 있다.

### 3.4 동적 Clarification 생성 기능
이 프로젝트는 고정된 입력 폼을 항상 띄우는 대신, 실제로 비어 있는 슬롯만 골라 동적으로 질문을 만든다. 구조가 없으면 구조를 묻고, 오비탈 종류가 없으면 HOMO/LUMO를 묻고, 모호성이 높으면 후보 중 하나를 선택하게 하는 식이다. 이 기능은 사용자의 입력 부담을 최소화하면서도 계산 실패를 줄이는 핵심 UX 장치이며, “자유입력 기반 도구인데도 안전하게 실행된다”는 인상을 만든다.

### 3.5 Clarification 응답 병합 기능
사용자가 clarification 폼에 응답하면 서버는 그 값을 기존 plan에 병합하고, 여전히 비어 있는 슬롯이 남아 있는지 다시 평가한다. 이 기능은 단순히 답변을 다시 처음부터 파싱하는 것이 아니라, 이미 확보한 문맥을 유지한 채 부족한 정보만 채워 넣는 방식이라 안정적이다. 덕분에 중복 질문이 줄고, 사용자는 시스템이 “내 이전 말을 기억한다”는 일관성을 체감하게 된다.

### 3.6 후속 분자 추론 기능
채팅 흐름에는 사용자의 후속 발화에서 분자 후보를 따로 추론하는 보조 기능이 있다. 이는 사용자가 길게 다시 설명하지 않고 “그럼 물로” 혹은 “벤젠 말고 아세톤”처럼 짧게 답해도 시스템이 기존 요청과 연결해 이해하도록 돕는다. 인간 대화의 자연스러운 축약 패턴을 흡수하는 기능이라서, 실제 사용 감각을 크게 개선한다.

### 3.7 채팅 결과 요약 기능
계산이 끝난 뒤 채팅 라우트는 결과 전체를 그대로 던지는 대신, 적절한 요약 문장을 생성해 대화 맥락에서 다시 제시한다. 현재는 `explanation.summary`가 있으면 그 요약을 우선 사용하고, 없으면 사람이 읽을 수 있는 기본 요약으로 폴백한다. 이 기능 덕분에 채팅 UI는 계산 API의 덤프를 노출하는 공간이 아니라, 계산 결과를 해석해 주는 대화 공간으로 작동한다.

### 3.8 채팅 헬스 및 메트릭 노출 기능
`/chat/health`는 단순한 alive 응답을 넘어서 현재 메트릭 요약을 같이 보여 준다. 여기에는 planner, resolver, compute, advisor 관련 성공/실패 카운터가 반영되므로, 채팅 표면에서도 시스템 상태를 운영 관점에서 살펴볼 수 있다. 개발 단계에서는 디버깅에, 운영 단계에서는 이상 징후 조기 감지에 유용한 기능이다.

## 4. 계산 요청과 잡 관리 기능

### 4.1 계산 요청 전처리 기능
`compute` 라우트는 계산을 바로 실행하기 전에 메시지에서 XYZ 블록을 추출하고, 자연어를 LLM 또는 휴리스틱으로 plan으로 바꾸고, 그 plan을 실제 실행 payload에 병합하는 전처리 계층을 갖고 있다. 이 기능은 사용자가 어떤 형식으로 입력하든 내부 계산기에는 일관된 구조를 넘기기 위한 핵심 단계다. 입력 다양성을 감싸고 계산기를 정형화된 호출 규약 뒤에 두는 역할을 한다.

### 4.2 계획 기반 잡 유형 결정 기능
입력에서 `intent`가 나오면 이를 `job_type`으로 매핑해 실제 계산 함수로 연결하는 기능이 있다. 사용자가 “ESP”, “partial charge”, “optimize”, “geometry”, “HOMO”처럼 서로 다른 표현을 써도, 시스템은 이를 표준화된 실행 유형으로 정리한다. 이 덕분에 프론트, LLM, 테스트, 계산기 사이에 같은 분류 체계를 공유할 수 있다.

### 4.3 계산 잡 제출 기능
`POST /compute/jobs`는 계산 요청을 받아 비동기 잡으로 큐에 올리는 기능을 제공한다. 이 기능의 핵심은 사용자가 긴 계산을 기다리며 HTTP 연결을 붙들고 있지 않아도 된다는 점이다. 서버는 잡 ID를 즉시 반환하고, 이후 사용자는 상태 조회, 이벤트 조회, 결과 조회를 분리해서 사용할 수 있어 장시간 작업을 더 안정적으로 관리할 수 있다.

### 4.4 동기 대기 실행 기능
같은 계산 엔드포인트는 `wait`, `sync`, `wait_for_result` 옵션을 통해 짧은 계산은 즉시 기다려서 결과까지 한 번에 돌려줄 수 있다. 이 기능은 데모, 테스트, 간단한 요청에 특히 유리하다. 비동기 기반 시스템이면서도 동기 UX를 일부 흉내 낼 수 있게 해 주므로, 도구를 쓰는 방식이 훨씬 유연해진다.

### 4.5 잡 목록 조회 기능
`GET /compute/jobs`는 현재까지 제출된 계산 잡들을 목록으로 보여 준다. 이 기능은 단지 디버깅용이 아니라 사용자가 자신의 계산 이력을 훑고, 어떤 작업이 진행 중인지, 무엇이 완료되었는지, 어떤 에러가 있었는지 파악하는 관리 화면의 기반 역할을 한다. 프론트엔드 히스토리 패널도 결국 이 구조를 활용해 상태를 조직한다.

### 4.6 개별 잡 상세 조회 기능
`GET /compute/jobs/{job_id}`는 특정 잡의 상태, 메시지, 진행률, 결과 포함 여부 등을 세부적으로 반환한다. 이 기능은 긴 계산을 폴링으로 추적할 때 중요하며, WebSocket을 사용하지 않는 클라이언트도 안정적으로 결과를 회수할 수 있게 한다. API 설계 관점에서 보면 이 프로젝트가 프론트 전용 앱이 아니라 외부 소비자도 염두에 둔 계산 서비스라는 점을 보여 준다.

### 4.7 잡 결과 전용 조회 기능
`GET /compute/jobs/{job_id}/result`는 결과만 분리해서 가져오도록 한 엔드포인트다. 이 기능은 상태와 결과를 분리해 다루고 싶은 외부 클라이언트에 편리하다. 특히 결과 payload가 크고 상태 폴링은 별도로 하고 싶을 때, 최소한의 응답만 받는 구조를 만들 수 있다는 점에서 API 사용성이 좋다.

### 4.8 잡 이벤트 조회 기능
`GET /compute/jobs/{job_id}/events`는 progress, info, completion 같은 이벤트 이력을 제공한다. 이 기능은 사용자가 “어디서 시간이 걸렸는지”를 뒤늦게라도 되짚어 볼 수 있게 해 주며, 디버깅과 UI 복원에도 유용하다. 단순 퍼센트 하나보다 훨씬 풍부한 실행 맥락을 제공하는 기능이다.

### 4.9 잡 삭제 기능
완료되었거나 실패한 잡을 삭제하는 기능은 서버 메모리와 사용자 작업공간을 정리하는 역할을 한다. 계산을 많이 수행하는 환경에서는 오래된 결과가 계속 쌓여 오히려 UI와 운영 가시성을 해칠 수 있는데, 이 기능이 정리 수단을 제공한다. 삭제는 비파괴적 실행 흐름 위에 붙은 유지보수 기능이라고 볼 수 있다.

### 4.10 잡 취소 기능
`POST /compute/jobs/{job_id}/cancel`은 장시간 계산이 필요 없어진 경우 즉시 취소할 수 있는 기능이다. 사용자가 잘못된 구조나 부적절한 방법으로 계산을 시작했을 때, 이 기능이 없다면 시스템 자원을 끝까지 태워야 한다. 진행 중 취소가 가능하다는 사실은 실제 서비스 체감에서 매우 중요하며, 계산 도구가 단순 스크립트가 아니라 상호작용형 시스템이라는 점을 강화한다.

### 4.11 선택 오비탈 큐브 재생성 기능
`POST /compute/jobs/{job_id}/orbital_cube`는 이미 계산된 잡을 바탕으로 다른 오비탈의 큐브 데이터를 다시 뽑아내는 기능이다. 이것은 아주 실용적인 기능인데, 사용자가 처음에는 HOMO를 보고 나중에 LUMO나 다른 MO를 더 보고 싶을 때 전체 계산을 다시 돌리지 않고도 후속 시각화가 가능해진다. 즉 계산과 시각화를 부분적으로 분리해 재사용성을 높여 준다.

### 4.12 인메모리 잡 매니저 기능
현재 웹 경로는 `InMemoryJobManager`를 사용해 잡 상태, 이벤트, 결과, 진행률을 관리한다. 이 기능은 복잡한 외부 큐 없이도 서버 단일 인스턴스에서 일관된 비동기 실행 경험을 제공하게 해 준다. 또한 event buffering과 terminal state 관리를 자체적으로 수행하므로, 프론트와의 상태 계약이 비교적 단순하게 유지된다.

### 4.13 잡 이력 디스크 저장 기능
인메모리 잡 매니저는 상태를 메모리에만 두지 않고 JSON 파일 형태의 히스토리로 저장하는 기능도 가진다. 이 기능은 서버 재시작 후에도 이전 잡에 대한 최소한의 흔적을 남길 수 있게 해 주며, 개발 단계의 디버깅에도 유용하다. 완전한 데이터베이스 영속성은 아니지만, lightweight한 운영 복원 기능으로는 충분히 의미가 있다.

## 5. 자연어 이해와 LLM 기능

### 5.1 QCVizAgent 계획 생성 기능
`QCVizAgent`는 자유로운 사용자 문장을 계산 가능한 plan 구조로 바꾸는 중심 역할을 한다. 이 기능은 단지 intent 분류기가 아니라, 구조명, 작업 종류, 메서드, basis, 전하, 다중도, 오비탈, ESP preset, clarification 필요 여부까지 함께 묶어 낸다. 결과적으로 이 에이전트는 “대화 입력을 계산 요청서로 변환하는 서기” 같은 역할을 한다.

### 5.2 Gemini 구조화 파서 기능
`services/gemini_agent.py`의 GeminiAgent는 Gemini 모델을 이용해 구조화된 JSON plan을 만들도록 설계되어 있다. 이 기능의 장점은 자유입력 해석 능력을 LLM에서 끌어오되, 최종 출력은 `PlanResponse` 스키마로 제한해 예측 불가능성을 줄인다는 점이다. 덕분에 모델을 쓰면서도 계산 시스템이 무너질 만큼 느슨한 free-form 응답에 의존하지 않게 된다.

### 5.3 Gemini 실패 시 휴리스틱 폴백 기능
LLM은 강력하지만 실패하거나 응답이 흔들릴 수 있으므로, 이 프로젝트는 Gemini 경로가 실패하면 규칙 기반 휴리스틱으로 내려가는 폴백 구조를 갖고 있다. 이 기능은 서비스 안정성 측면에서 매우 중요하다. 실제 API 키 오류, 네트워크 문제, 모델 응답 이상이 있어도 시스템 전체가 멈추지 않고, 최소한의 계산 요청 처리 능력을 유지할 수 있기 때문이다.

### 5.4 한국어 오타 및 혼합 입력 정규화 기능
`llm/normalizer.py`는 “무ㄹ오비탈 계산”, “아세톤최적화”, “emim tfsi esp”처럼 오타, 붙여쓰기, 한글/영문 혼용이 섞인 입력을 정규화하는 기능을 제공한다. 이 기능은 연구자나 학생이 실제로 쓰는 입력 습관에 가깝게 시스템을 맞춰 준다는 점에서 가치가 크다. 사람이 의도한 의미를 추정해 정리한 뒤에 계산기로 넘겨 주므로, 사용자에게는 도구가 훨씬 “영리하다”는 인상을 준다.

### 5.5 구조 후보 추출 기능
정규화 계층은 문장 전체에서 분자명 후보를 따로 뽑아내는 기능도 갖고 있다. 이 기능은 작업 키워드와 구조 키워드가 한 문장에 뒤섞여 있을 때 유용하며, “benzene HOMO”, “물 전하”, “acetone esp map” 같은 입력을 자연스럽게 분해할 수 있게 한다. 구조 추출이 안정되면 그 뒤의 resolver, MolChat, PubChem 파이프라인 성능도 같이 좋아진다.

### 5.6 작업 힌트 감지 기능
정규화는 구조뿐 아니라 “이 입력이 orbital인지, esp인지, optimize인지, explanation인지”에 대한 힌트도 뽑아낸다. 이 기능은 LLM이 없거나 불안정할 때도 어느 정도 의미 있는 plan을 만드는 데 기여한다. 즉 정규화 계층은 단순 문자열 청소기가 아니라 lightweight semantic pre-parser 역할을 한다.

### 5.7 계획 스키마 검증 기능
`llm/schemas.py`에 정의된 typed schema는 plan, clarification form, slot merge, result explanation 같은 주요 데이터 구조를 검증하는 기능을 제공한다. 이 기능은 프론트엔드, 라우트, LLM, 계산기 사이에 데이터 계약을 명시적으로 박아 넣는 역할을 한다. 시스템이 커질수록 “형태가 맞는 데이터만 통과시킨다”는 성질이 중요해지는데, 이 스키마 계층이 그 안전장치다.

### 5.8 결과 설명용 LLM 브리지 기능
`llm/bridge.py`는 단순 파서가 아니라 LLM을 advisor나 고수준 의도 흐름에 연결하는 브리지 기능을 제공한다. 이 기능은 사용자의 목적과 계산 도구를 연결하는 해석 계층으로 볼 수 있다. 나중에 더 복잡한 멀티턴 워크플로나 advisor orchestration을 강화할 때, 이 계층이 중심 허브 역할을 하게 된다.

## 6. 구조 해석과 외부 화학 서비스 기능

### 6.1 MolChat 우선 구조 해석 기능
이 프로젝트의 메인 구조 해석 파이프라인은 MolChat API를 1순위로 사용한다. 사용자가 “water”, “아세톤”, “benzene”처럼 이름 기반으로 입력하면, 먼저 MolChat이 이를 CID, 카드, SMILES, 3D SDF로 이어서 해석하고 최종적으로 XYZ 좌표까지 만드는 흐름이다. 이 기능은 단순 이름 매칭이 아니라 외부 분자 백엔드를 통해 실제 계산 가능한 3차원 구조를 확보한다는 점에서 핵심적이다.

### 6.2 PubChem 폴백 기능
MolChat에서 구조 해석이 실패할 경우 PubChem으로 내려가는 폴백 경로가 준비되어 있다. 이 기능은 외부 API 하나에 전체 기능을 걸지 않게 해 주며, 실제 서비스 안정성을 크게 높인다. 사용자는 내부에서 어떤 경로를 탔는지 몰라도, 시스템은 성공 확률을 높이기 위해 여러 공용 화학 데이터 소스를 단계적으로 활용한다.

### 6.3 한국어 별칭 번역 기능
`ko_aliases.py`는 한국어 분자명을 영어 또는 canonical query로 바꾸는 기능을 제공한다. 이 기능은 국내 사용자를 염두에 둔 실질적인 접근성 개선 장치다. 사용자는 “물”, “아세톤”, “벤젠”처럼 익숙한 표현으로 말해도 되고, 시스템은 이를 외부 화학 서비스가 이해할 수 있는 형태로 변환해 주기 때문에 학습 비용이 낮다.

### 6.4 약어 확장 기능
배터리 전해질과 용매에서 자주 등장하는 `EMIM`, `TFSI`, `EC`, `DMC`, `THF` 같은 약어를 풀어 canonical query 후보로 확장하는 기능이 들어 있다. 이 기능은 화학 도메인 특화 UX에 해당하며, 특히 이온성 액체나 전해질 연구 맥락에서 매우 유용하다. 일반적인 LLM 챗봇보다 더 “실험실 입력”에 가까운 표현을 이해하도록 만드는 요소다.

### 6.5 질의 계획(Query Plan) 생성 기능
StructureResolver는 단순히 원문 문자열 하나만 던지는 것이 아니라, `raw_query`, `translated_query`, `normalized_query`, `candidate_queries`로 이루어진 질의 계획을 만든다. 이 기능은 구조 해석이 한 번에 안 될 때도 여러 변형을 시도하게 해주며, 나중에 왜 특정 구조가 풀렸는지 추적 가능하게 한다. 디버깅과 성공률 향상 두 측면에서 모두 중요한 기능이다.

### 6.6 이온쌍 해석 기능
`ion_pair_handler.py`는 여러 이온 종이 함께 들어온 입력을 감지하고, 각 구성 요소를 해석한 뒤 하나의 이온쌍 구조 결과로 조합하는 기능을 제공한다. 이 기능은 단일 중성분자만 다루는 도구에서 한 단계 나아간 것이다. 실제 배터리 전해질이나 이온성 액체 연구에서는 이온쌍이나 복합계 입력이 많기 때문에, 이 기능은 프로젝트의 도메인 적합성을 크게 높인다.

### 6.7 SDF를 XYZ로 바꾸는 구조 변환 기능
MolChat이나 PubChem이 반환한 구조를 계산기에 넘기려면 XYZ 형식이 필요하기 때문에, `sdf_converter.py`는 SDF 텍스트를 XYZ 좌표로 바꾸는 기능을 수행한다. 이 기능은 외부 화학 서비스와 내부 PySCF 계산기 사이의 형식 차이를 메우는 변환 계층이다. 사용자 입장에서는 보이지 않지만, 시스템이 여러 서비스 사이를 유연하게 넘나들 수 있게 하는 기술적 접착제 역할을 한다.

## 7. 계산 엔진 기능

### 7.1 구조 확인 전용 계산 기능
`run_resolve_structure`는 실제 전자구조 계산을 하지 않고, 구조가 잘 해석되었는지와 기본 시각화가 가능한지 확인하는 경량 기능이다. 이 기능은 사용자가 먼저 구조가 맞는지 빠르게 점검하고 싶을 때 유용하며, 계산 비용을 들이기 전에 입력 단계에서 문제를 잡아낸다. 계산 워크플로우의 “preflight check”에 해당한다.

### 7.2 기하 정보 분석 기능
`run_geometry_analysis`는 원자 좌표를 바탕으로 원자 수, 결합 정보, 결합 길이 범위, 분자식 등 기하학적 요약을 반환하는 기능이다. 이 기능은 에너지를 구하지 않아도 구조 자체를 검토하고 싶은 상황에서 가치가 있다. 시각화와 함께 쓰면 “내가 의도한 구조가 정말 이런 형태인가”를 빠르게 검증할 수 있다.

### 7.3 단일점 에너지 계산 기능
`run_single_point`는 주어진 구조에서 한 번의 SCF 계산을 수행하여 총 에너지와 기본 전자구조 정보를 얻는 기능이다. 양자화학 워크플로우의 기본 축에 해당하며, 다른 고급 분석의 출발점 역할도 한다. 구조 최적화 없이 현재 좌표 기준의 전자 상태를 보고 싶을 때 가장 직접적인 계산 기능이다.

### 7.4 부분 전하 계산 기능
`run_partial_charges`는 Mulliken과 관련 전하 정보를 계산하여 각 원자에 대한 부분 전하 리스트를 제공한다. 이 기능은 반응성 직관, 친전자성/친핵성 해석, 분자 내 전하 분포 파악에 유용하다. 정밀한 전하 모델의 철학적 한계는 있지만, 빠른 탐색과 교육적 설명에는 매우 실용적인 기능이다.

### 7.5 오비탈 프리뷰 기능
`run_orbital_preview`는 선택한 오비탈의 에너지, 점유수, 인덱스, 큐브 데이터 등을 계산해 3D 시각화 가능한 형태로 반환한다. 이 기능은 사용자가 가장 직관적으로 “양자화학 같다”고 느끼는 지점 중 하나로, HOMO/LUMO 같은 frontier orbital을 눈으로 확인하게 해 준다. 시각적으로도 강한 기능이라 데모 효과가 매우 좋다.

### 7.6 ESP 맵 계산 기능
`run_esp_map`은 전자밀도와 전기정전위(ESP) 관련 큐브 데이터를 생성하고, 자동 범위 추정과 함께 색상 매핑 가능한 결과를 돌려준다. 이 기능은 반응 부위 추론, 표면 전하 분포 해석, 직관적 시각화에 큰 장점이 있다. 특히 이 프로젝트는 자동 symmetric range 추정까지 포함하고 있어, 사용자가 수동으로 스케일을 맞추지 않아도 보기 좋은 결과를 얻을 수 있다.

### 7.7 기하 최적화 기능
`run_geometry_optimization`은 초기 구조를 최적화한 뒤 최종 SCF를 다시 수행하여 안정화된 구조와 에너지 정보를 제공한다. 이 기능은 단순한 정적 분석이 아니라 실제 계산 화학 워크플로우의 핵심 단계 중 하나다. 구조를 손으로 추정해 넣은 뒤, 그 구조가 더 안정한 방향으로 어떻게 움직이는지 확인할 수 있다는 점에서 실험적인 가치가 크다.

### 7.8 종합 분석 기능
`run_analyze`는 단일점, 오비탈, 전하, ESP, 기하 정보 등 여러 관측량을 한 번에 묶어내는 포괄 기능이다. 이 기능은 사용자가 “전체적으로 한 번 보고 싶다”는 요구를 할 때 잘 맞는다. 개별 계산을 여러 번 호출하지 않고도 폭넓은 요약을 만들 수 있어, 탐색적 분석의 첫 단계로 특히 유용하다.

### 7.9 SCF 수렴 추적 기능
계산 엔진은 SCF 사이클마다 에너지와 dE를 수집해 `scf_history`, `n_scf_cycles`, `scf_elapsed_s`, `scf_final_delta_e_hartree`로 내보낸다. 이 기능은 단순히 결과값만 주는 계산기보다 훨씬 설명력이 크다. 사용자는 계산이 얼마나 빨리 수렴했는지, 어느 정도 안정적으로 끝났는지, 혹은 애매한 수렴 상태였는지를 시각적으로 판단할 수 있다.

### 7.10 실시간 진행 이벤트 방출 기능
PySCF 러너는 resolve, build_scf, scf, orbital_cube, charges, optimize, done 같은 단계별 progress 이벤트를 외부 콜백으로 내보낸다. 이 기능이 있기 때문에 채팅 UI와 잡 관리자에서 실제 진행률 막대와 실시간 상태 메시지를 표현할 수 있다. 계산 코어가 UI와 느슨하게 결합되어 있으면서도 충분한 피드백을 제공한다는 점이 강점이다.

### 7.11 디스크 캐시 기반 SCF 재사용 기능
`disk_cache.py`와 러너 내부 캐시 키 계산은 동일 구조, 방법, basis, 전하, 다중도 조합에 대해 SCF 결과를 재사용하는 기능을 제공한다. 이 기능은 반복 실험이나 같은 분자를 여러 방식으로 후처리할 때 시간 절감 효과가 크다. 특히 교육/데모 환경에서 체감 속도를 높이는 데 도움이 된다.

### 7.12 안전성 평가 기능
`safety_guard.py`는 원자 수, basis, 메모리 추정 등을 바탕으로 요청이 너무 무겁거나 위험한지 평가하는 기능을 제공한다. 이 기능은 단순 보호장치 같아 보여도, 실제 서비스에서는 서버를 살리는 역할을 한다. 사용자가 무심코 지나치게 큰 계산을 던졌을 때 시스템이 이를 감지하고 제동을 걸 수 있기 때문이다.

## 8. 결과 후처리와 해설 기능

### 8.1 결과 계약 정규화 기능
계산 함수가 반환하는 결과는 모두 `compute` 라우트에서 일정한 계약 형태로 정규화된다. 이 기능은 프론트엔드가 job type마다 완전히 다른 응답 형태를 감당하지 않아도 되게 해 주며, 결과 소비 계층을 단순화한다. 결과적으로 계산기 내부가 바뀌더라도 외부 계약은 비교적 안정적으로 유지된다.

### 8.2 결과 초점 탭 결정 기능
결과를 받은 뒤 어떤 탭을 우선 보여 줄지 자동으로 결정하는 기능이 있다. 오비탈 계산이면 orbital 탭, ESP 계산이면 esp 탭, 전하 계산이면 charges 탭을 우선 포커스하는 식이다. 이 기능은 작지만 UX 영향이 크며, 사용자가 결과를 받자마자 가장 관련 높은 뷰를 보게 해준다.

### 8.3 결과 해설 생성 기능
`web/result_explainer.py`는 계산 결과를 바탕으로 summary, key findings, interpretation, cautions, next actions를 생성하는 기능을 제공한다. 이 기능 덕분에 수치와 큐브 파일만 있는 결과가 사람이 읽는 설명으로 바뀐다. 특히 초보 사용자는 숫자만 던져졌을 때 무엇을 봐야 할지 모르기 쉬운데, 이 해설 계층이 결과를 “읽을 수 있는 이야기”로 바꿔 준다.

### 8.4 다음 액션 추천 기능
결과 해설은 단순 해설에 그치지 않고 “다음으로 무엇을 해보면 좋은지”까지 제안한다. 예를 들어 오비탈 계산 후 basis를 키워 보거나, ESP 결과 후 부분 전하 비교를 권하거나, 최적화 후 단일점 재계산을 제안하는 식이다. 이 기능은 사용자의 흐름을 한 번의 계산에서 끝내지 않고, 더 깊은 탐색으로 자연스럽게 이끈다.

### 8.5 Advisor 후처리 통합 기능
계산 결과에는 advisor payload와 advisor_summary를 붙여서, 단순 계산 엔진과 연구 보조 기능을 하나의 응답으로 묶는다. 이 기능은 계산 도구와 보고서/방법론 조언 도구를 분리하지 않고 하나의 워크플로우로 통합한다는 점에서 의미가 크다. 사용자는 계산이 끝난 직후 “이 설정이 적절했는지”에 대한 메타 정보를 곧바로 함께 받는다.

## 9. Advisor 기능

### 9.1 계산 프리셋 추천 기능
Advisor의 대표 기능은 분자 구조와 목표를 보고 functional, basis, 전략을 추천하는 프리셋 추천이다. 이 기능은 초보 사용자가 방법론 선택에서 막히는 지점을 줄여 준다. 단순 자동완성이 아니라 분자의 크기, 목적, 비용 균형을 고려해 “왜 이 방법이 어울리는지”까지 설명하는 쪽으로 확장될 수 있는 토대다.

### 9.2 Methods 섹션 초안 생성 기능
`methods_drafter.py`와 대응 tool은 계산 내용을 논문이나 보고서의 Methods 섹션 문안으로 바꾸는 기능을 갖고 있다. 이 기능은 연구자 관점에서 매우 실용적이다. 계산은 끝났지만 문서화가 번거로운 상황에서, 적어도 구조화된 초안을 제공해 작성 시간을 줄일 수 있다.

### 9.3 재현 스크립트 생성 기능
`script_generator.py`는 계산을 재현할 수 있는 스크립트 초안을 만들어 주는 기능이다. 이는 재현성 측면에서 중요하다. 사용자에게 계산 결과뿐 아니라 “이 결과를 다시 만들 수 있는 실행 레시피”를 제공함으로써, 계산이 일회성 클릭 이벤트로 끝나지 않게 한다.

### 9.4 문헌 검증 기능
`literature_validator.py`는 계산 결과의 일부를 문헌이나 기준 데이터와 비교해 타당성을 가늠하는 기능을 제공한다. 이 기능은 시스템이 결과를 맹목적으로 포장하지 않도록 만드는 제동장치 역할을 한다. 계산값이 문헌 경향과 크게 어긋나는지, 적어도 상식적인 범위 안에 있는지 판단할 수 있게 해 준다.

### 9.5 신뢰도 점수 산정 기능
`confidence_scorer.py`는 방법론, 구조, 문헌 일치도, 입력 품질 같은 요소를 바탕으로 계산 결과의 신뢰도 레벨을 평가한다. 이 기능은 결과를 절대적 진실처럼 보이게 하지 않고, 어느 정도 신뢰하고 해석해야 하는지를 정량적 힌트로 제공한다. 사용자에게 책임 있는 피드백을 주기 위한 기능이라고 볼 수 있다.

### 9.6 Geometry 기반 advisor 진입 기능
최근 추가된 `prepare_advisor_plan_from_geometry`는 이미 해석된 구조나 XYZ를 기준으로 advisor를 실행할 수 있게 하는 기능이다. 이 기능은 advisor가 반드시 레거시 resolver를 다시 타지 않아도 된다는 뜻이며, 웹 메인 플로우와 advisor를 더 자연스럽게 접합한다. 즉 실제 계산 직전 또는 직후에 구조를 재활용해 조언을 제공할 수 있게 된다.

### 9.7 Advisor 요약 압축 기능
`summarize_advisor_payload`는 advisor의 복잡한 결과를 프론트에 바로 쓰기 좋은 compact summary로 바꿔 주는 기능이다. 이 기능은 풍부한 내부 결과와 간결한 UI 사이를 연결한다. 사용자는 복잡한 raw payload를 볼 필요 없이 추천 functional, basis, confidence, methods preview 같은 핵심만 요약해서 받을 수 있다.

## 10. 프론트엔드 사용자 경험 기능

### 10.1 전역 앱 상태 저장 기능
`index.html`에 부트스트랩되는 `QCVizApp` 전역 스토어는 잡, 결과, 활성 탭, 채팅 메시지, 세션 ID, 테마, UI snapshot을 통합 관리한다. 이 기능 덕분에 각 프론트 모듈이 서로 분리되어 있으면서도 같은 상태를 공유할 수 있다. 작은 규모의 프론트 상태관리 라이브러리를 자체 구현한 셈이라, 의존성 없이도 앱 일관성을 유지한다.

### 10.2 결과 히스토리 보관 기능
프론트는 잡 결과를 히스토리에 누적 보관하고, 사용자가 이전 결과를 다시 열람할 수 있게 한다. 이 기능은 계산 도구를 “한 번 쓰고 끝나는 화면”이 아니라 작업 기록이 남는 환경으로 바꿔 준다. 여러 분자나 여러 계산을 비교하며 작업하는 사용자에게 특히 중요하다.

### 10.3 채팅 메시지 로컬 저장 기능
채팅 메시지는 localStorage에 저장되어 새로고침 후에도 일부 복원된다. 이 기능은 장시간 실험하거나 브라우저를 다시 열 때 흐름이 완전히 사라지지 않도록 도와준다. 완전한 서버 저장형 대화 이력은 아니지만, 일상적인 사용 경험을 훨씬 안정적으로 만든다.

### 10.4 동적 Clarification 폼 렌더링 기능
프론트 `chat.js`는 서버가 내려준 `ClarificationForm` 스키마를 받아 `text`, `textarea`, `radio`, `select`, `number`, `checkbox` 컴포넌트로 렌더링한다. 이 기능은 폼 자체를 서버가 HTML로 생성하지 않고도 매우 유연한 질문 UX를 만든다. 결과적으로 프론트는 안전한 렌더러 역할만 하고, 질문의 의미는 서버/LLM이 결정하는 구조가 완성된다.

### 10.5 실시간 진행률 바 기능
채팅 UI는 진행 중인 계산에 대해 progress bar와 단계별 메시지를 표시한다. 이 기능은 계산이 오래 걸릴 때 사용자가 시스템이 멈춘 것으로 오해하지 않게 해 주며, “현재 구조 해석 중인지, SCF 중인지, 큐브 생성 중인지”를 직관적으로 알려 준다. 긴 계산에서 사용자 불안을 크게 줄여 준다.

### 10.6 진행 중 SCF 수렴 미니 차트 기능
진행률 카드 내부에는 SCF dE 이력을 바탕으로 간단한 수렴 차트를 그리는 기능이 있다. 이 기능은 계산이 끝난 뒤 결과 화면에서만 보는 것이 아니라, 계산이 도는 도중에도 수렴 양상을 볼 수 있게 해 준다. 사용자는 단순히 퍼센트가 아니라 계산이 “잘 되고 있는지”를 더 실제적으로 파악할 수 있다.

### 10.7 결과 탭 분리 기능
결과 패널은 summary, geometry, orbital, esp, charges, json 탭으로 나뉘어 각 관점을 별도로 보여 준다. 이 기능은 복합 결과를 한 화면에 억지로 우겨넣지 않고, 사용자가 관심 관점에 따라 결과를 탐색하게 만든다. 정보 밀도가 높은 과학 계산 결과를 UI 차원에서 정리하는 중요한 구조다.

### 10.8 오비탈/에너지 단위 토글 기능
결과 UI는 HOMO, LUMO, gap, total energy 같은 수치를 eV, Hartree, kcal/mol 사이에서 바꿔 볼 수 있게 한다. 이 기능은 계산 화학 사용자마다 익숙한 단위가 다르다는 현실을 반영한 것이다. 같은 결과를 여러 단위로 해석할 수 있게 해 주므로, 연구자와 학생 모두에게 더 친화적이다.

### 10.9 ESP 범위 단위 토글 기능
ESP 자동 범위나 통계 역시 atomic unit와 kcal/mol 단위로 오가며 볼 수 있게 구현되어 있다. 이 기능은 단순 장식이 아니라, ESP를 해석하는 맥락을 사용자가 선택하게 해 준다. 어떤 사용자는 계산 내부 단위에 익숙하고, 어떤 사용자는 보다 화학적으로 읽기 쉬운 단위를 선호하므로 둘 다 지원하는 것이 유리하다.

### 10.10 결과 해설 블록 표시 기능
summary 탭은 단순 수치 테이블이 아니라 해설 블록을 함께 표시해, 결과의 의미를 서술형으로 다시 전달한다. 이 기능은 특히 입문자나 비전문가에게 중요하며, 결과를 “읽는 화면”으로 만든다. 고급 사용자에게도 빠른 1차 요약을 제공한다는 점에서 가치가 있다.

### 10.11 Advisor 요약 블록 표시 기능
summary 탭에는 advisor에서 생성한 추천 정보도 별도 블록으로 나타난다. 이 기능은 계산과 연구 조언 사이의 간격을 줄이며, 사용자가 별도 메뉴로 이동하지 않아도 핵심 방법론 힌트를 바로 확인하게 해 준다. 계산 결과 화면이 곧 decision support 화면이 되도록 만드는 기능이다.

### 10.12 3D Viewer 연동 기능
프론트에는 분자 구조, 오비탈, ESP를 보여 주는 viewer 계층이 있어, 결과 데이터를 즉시 시각적으로 탐색할 수 있다. 이 기능은 숫자와 좌표를 3차원 감각으로 바꿔 주는 핵심 요소다. 양자화학 결과는 텍스트만으로 전달하기 어렵기 때문에, viewer는 이 프로젝트의 인상 자체를 좌우하는 중심 기능 중 하나다.

### 10.13 테마 및 상태 표시 기능
UI는 테마 상태, WebSocket 연결 상태, 현재 시스템 상태 메시지를 별도로 관리한다. 이 기능은 앱이 단순 계산 폼이 아니라 지속적인 상태를 가진 작업 환경이라는 느낌을 만든다. 연결 여부, 처리 상태, 활성 작업이 보이는 것은 사용자가 시스템을 신뢰하는 데 중요한 요소다.

## 11. 시각화와 렌더링 백엔드 기능

### 11.1 웹 시각화 payload 생성 기능
`backends/viz_backend.py`는 계산 결과를 웹 프론트가 바로 먹을 수 있는 visualization payload로 변환하는 기능을 제공한다. 이 기능은 계산기와 UI 사이의 번역기 역할을 한다. 큐브 데이터, XYZ, 기본 스타일, ESP 설정, 탭 활성화 정보를 함께 정리해 주므로, 프론트는 결과를 해석하기보다 렌더링에 집중할 수 있다.

### 11.2 큐브 데이터 정규화 기능
`CubeNormalizer` 같은 계층은 큐브 형식이나 시각화 입력을 일정한 형식으로 다듬어, 서로 다른 생성 경로에서 온 데이터를 같은 방법으로 처리하게 해 준다. 이 기능은 시각화 품질의 일관성에 중요하다. 데이터 형식 차이를 프론트가 감당하지 않아도 되게 해 주기 때문이다.

### 11.3 Py3Dmol 기반 브라우저 시각화 기능
`Py3DmolBackend`와 연결된 웹 시각화 계층은 브라우저 안에서 분자 구조와 표면을 인터랙티브하게 보여 주는 기능을 담당한다. 이 기능은 설치형 과학 소프트웨어가 아니라 웹 브라우저만으로도 시각적 분석을 가능하게 만든다. 연구 결과를 공유하거나 교육용으로 시연할 때 특히 유리하다.

### 11.4 PyVista 렌더링 기능
`renderers/pyvista_renderer.py`는 큐브 데이터를 오프라인 렌더링 이미지로 만드는 기능을 제공한다. 이 기능은 브라우저 인터랙션이 아닌 정적 이미지 산출이 필요한 상황에 적합하다. 예를 들어 보고서, 문서, 썸네일 생성 같은 시나리오에서 재사용 가치가 있다.

### 11.5 HTML to PNG 내보내기 기능
`png_exporter.py`는 Playwright 기반으로 HTML 시각화를 PNG로 캡처하는 기능을 제공한다. 이 기능은 웹에서 본 화면을 그대로 결과물로 남기고 싶은 경우 유용하다. 대화형 UI를 정적 산출물로 바꾸는 브리지 역할을 한다는 점에서 실무적 가치가 있다.

### 11.6 최적 렌더러 자동 선택 기능
`renderers/__init__.py`와 `tools/health.py`는 현재 환경에서 쓸 수 있는 렌더러를 감지하고 가장 적합한 경로를 선택하는 기능을 가진다. 이 기능 덕분에 환경마다 다른 설치 상태를 모두 수동으로 신경 쓰지 않아도 된다. 즉 개발용 PC, 서버, 헤드리스 환경 등에서 가능한 옵션을 자동으로 고르는 유연성이 생긴다.

## 12. MCP 및 도구 기능

### 12.1 FastMCP 서버 등록 기능
`mcp_server.py`는 FastMCP 서버를 초기화하고 core/advisor 도구를 등록하는 기능을 수행한다. 이 기능은 이 프로젝트를 단순 웹앱이 아니라 AI 에이전트가 호출할 수 있는 툴 서버로 바꾼다. 향후 다른 에이전트 시스템이 “분자 계산”, “ESP 생성”, “advisor 조언”을 호출 가능한 도구처럼 사용할 수 있게 된다.

### 12.2 레거시 분자 resolver 도구 기능
`tools/core.py`의 `MoleculeResolver`는 웹 메인 구조 해석 경로와 별도로, 도구 계층에서 작동하는 resolver 기능을 제공한다. 이 기능은 현재 과도기적 성격이 있지만, MCP나 레거시 경로에서 여전히 의미가 있다. 웹 메인 경로와 완전히 같지는 않아도, 외부 툴 인터페이스에서 구조를 해석하고 계산으로 이어 주는 역할을 맡는다.

### 12.3 IBO 계산 도구 기능
MCP core에는 IBO 계산을 수행하는 기능이 포함되어 있어, 단순 HOMO/LUMO 프리뷰보다 더 고급 결합 해석 쪽으로 확장할 수 있다. 이 기능은 현재 웹 주경로보다 에이전트/도구 경로에서 더 두드러지지만, 프로젝트의 분석 범위를 넓혀 주는 중요한 자산이다. 양자화학 시각화가 단순 표면 렌더링에 머물지 않음을 보여 준다.

### 12.4 ESP 계산 도구 기능
core tool의 `compute_esp`는 웹 경로 바깥에서도 전기정전위 계산을 호출할 수 있게 해 준다. 이 기능은 브라우저 없이도 에이전트나 스크립트가 ESP 관련 작업을 수행하게 해 준다는 점에서 의미가 있다. 웹과 MCP가 같은 화학 기능을 서로 다른 소비 방식으로 노출하고 있다는 예다.

### 12.5 부분 전하 도구 기능
`compute_partial_charges` tool은 전하 계산을 독립 도구처럼 사용할 수 있게 한다. 이 기능은 작은 분석 작업을 자동화 스크립트에 끼워 넣거나, 다른 에이전트가 결과를 후속 reasoning에 활용하게 만들기 좋다. 사용자 인터페이스를 벗어난 실행형 구성 요소로서 가치가 있다.

### 12.6 오비탈 시각화 도구 기능
`visualize_orbital` tool은 선택 오비탈을 시각화 가능한 결과로 만드는 기능이다. 이 기능은 웹 UI의 시각화 경험을 MCP/도구 세계로 가져온 셈이다. 즉 브라우저가 없어도 에이전트는 오비탈 표현을 요청하고, 그 결과를 다른 문서 생성이나 분석 흐름에 쓸 수 있다.

### 12.7 형식 변환 도구 기능
`convert_format`과 관련 로직은 입력 파일이나 산출물의 형식을 바꾸는 실용 기능을 제공한다. 이 기능은 계산 화학 작업에서 자주 생기는 “형식 장벽”을 낮춘다. 프로젝트가 단순 계산 실행기에서 실제 연구 도구 상자 역할로 확장되는 데 필요한 기능이다.

### 12.8 결합 분석 도구 기능
`analyze_bonding`은 분자의 결합 특성을 분석하는 기능으로, 구조 시각화와 전자구조 계산 사이를 더 깊게 연결해 준다. 이 기능은 프로젝트가 단순 그림 생성기가 아니라 결합과 전자분포 자체를 읽는 도구가 되도록 돕는다. 교육적·연구적 확장 여지가 큰 기능이다.

### 12.9 도구 헬스체크 기능
`tools/health.py`는 렌더러, 패키지, 메트릭 상태를 포함한 도구 수준의 헬스 정보를 제공한다. 이 기능은 특히 에이전트 환경에서 “현재 서버가 어떤 능력을 가지고 있는지”를 외부에 알려 주는 역할을 한다. 도구를 호출하기 전에 환경 적합성을 확인하는 셀프 설명 기능으로 유용하다.

## 13. 분석 보조 기능

### 13.1 XYZ sanitize 기능
`analysis/sanitize.py`는 사용자가 준 XYZ나 atom spec을 정리하고, 과도한 입력이나 깨진 형식을 안전한 형태로 다듬는 기능을 제공한다. 이 기능은 계산기의 입력 안정성을 높이고, 엉뚱한 문자열 때문에 PySCF가 깊숙한 곳에서 실패하는 상황을 줄여 준다. 입력 위생을 관리하는 기초 체력 같은 기능이다.

### 13.2 분자 조각 감지 기능
`fragment_detector.py`는 구조 내에서 fragment를 감지하고 요약하는 기능을 제공한다. 이 기능은 복합계나 이온쌍, 비공유 조합 구조를 다룰 때 특히 의미가 있다. 구조를 단순 원자 리스트가 아니라 조각들의 집합으로 보는 시각을 제공한다는 점에서 해석 폭을 넓혀 준다.

### 13.3 조각 간 전하 이동 해석 기능
`charge_transfer.py`는 fragment 단위 전하를 계산하고 전하 이동 방향을 해석하는 기능을 제공한다. 이 기능은 단순 원자별 전하보다 더 개념적인 해석을 가능하게 만든다. 도너-억셉터 성격이나 조각 간 전하 재분배를 이해하는 데 도움이 된다.

## 14. 백엔드 추상화와 확장 기능

### 14.1 Backend 추상 인터페이스 기능
`backends/base.py`는 OrbitalBackend, ParserBackend, VisualizationBackend, StructureBackend 같은 추상 인터페이스를 정의한다. 이 기능은 프로젝트를 특정 구현체에 완전히 묶지 않게 한다. 장기적으로 PySCF 말고도 다른 계산기나 파서를 교체·추가할 수 있는 구조적 유연성을 준다.

### 14.2 PySCFBackend 확장 기능
`backends/pyscf_backend.py`는 웹 러너 외의 더 일반화된 PySCF 백엔드 구현을 제공한다. 이 기능은 계산 엔진을 더 모듈화된 형태로 다루고자 할 때 중요하다. 또한 수렴 전략, 큐브 파싱, CLI 스타일 리포팅 같은 기능을 포함해 확장성 있는 계층을 마련한다.

### 14.3 ASE 구조 백엔드 기능
`ASEBackend`는 구조 계층에서 ASE를 활용할 수 있는 확장 포인트를 제공한다. 이 기능은 구조 조작과 변환, 외부 포맷 연계에서 향후 확장 여지를 남긴다. 현재 메인 플로우의 핵심은 아니더라도, 화학/재료 계산 생태계와의 연결성을 높이는 자산이다.

### 14.4 cclib 파싱 백엔드 기능
`CclibBackend`는 외부 계산 출력 파일을 파싱해 구조화된 결과로 읽어들이는 방향의 확장 기능이다. 이 기능은 프로젝트가 “자체 계산만” 다루는 것이 아니라, 외부 계산 결과를 가져와 시각화하고 분석하는 플랫폼으로도 발전할 수 있게 한다. 연구 현장에서는 이미 존재하는 출력 파일을 재활용하는 경우가 많기 때문에 잠재 가치가 높다.

### 14.5 백엔드 레지스트리 기능
`registry.py`는 사용 가능한 백엔드를 등록하고 조회하는 기능을 제공한다. 이 기능은 프로젝트가 다중 구현체를 품을 때 필수적인 구조다. 어떤 기능을 어떤 구현체가 담당하는지 명시적으로 관리할 수 있어 유지보수성이 좋아진다.

## 15. 실행, 관측성, 설정 기능

### 15.1 경량 실행 캐시 기능
`execution/cache.py`와 advisor 쪽 execution cache는 반복 계산이나 중간 결과를 가볍게 재사용하는 기능을 제공한다. 이 기능은 무거운 계산 환경에서 특히 유용하며, 불필요한 중복 실행을 줄여 전체 응답성을 개선한다. 프로젝트가 조금씩 “서비스 운영”을 염두에 두고 있음을 보여 주는 부분이다.

### 15.2 워커 기반 실행 기능
`execution/worker.py`와 advisor execution worker는 시간이 걸리는 작업을 메인 흐름과 분리해 다루는 기반을 제공한다. 이 기능은 확장 단계에서 장시간 LLM 작업이나 advisor 후처리를 별도 워커로 넘기는 데 도움이 된다. 현재는 전면적으로 쓰이지 않아도, 아키텍처적으로는 확장성 포인트다.

### 15.3 관측성 메트릭 수집 기능
`observability.py`의 `track_operation`과 `MetricsCollector`는 planner, resolver, compute, advisor 같은 경로의 성공/실패와 수행 시간을 수집하는 기능을 제공한다. 이 기능은 시스템을 “돌아가는 코드”에서 “관측 가능한 서비스”로 끌어올린다. 성능 저하나 실패율 상승을 감지하는 최소한의 운영 감각이 이미 들어와 있다는 뜻이다.

### 15.4 웹 메트릭 요약 노출 기능
`compute_health`와 `chat_health`는 내부 카운터를 사람이 읽기 쉬운 success ratio 형태로 가공해 내보낸다. 이 기능 덕분에 운영자는 raw counter를 따로 계산하지 않아도 planner_success_ratio, resolver_success_ratio 같은 핵심 지표를 바로 볼 수 있다. 작은 프로젝트라도 서비스답게 운영하려는 의도가 반영된 기능이다.

### 15.5 설정 로딩 기능
`config.py`와 환경변수 기반 설정은 API 키, 모델명, 캐시 크기, 워커 수, 포트 성격의 변수를 코드와 분리해 관리하는 기능을 제공한다. 이 기능은 같은 코드베이스를 로컬, 개발, 테스트, 운영 환경에서 다르게 움직이게 하는 기본 장치다. LLM provider나 MolChat, 캐시 전략을 실험적으로 바꾸기에도 유리하다.

### 15.6 로깅 설정 기능
`log_config.py`와 각 모듈 로거는 실행 흐름을 추적하고 문제를 진단하는 데 필요한 로그 체계를 제공한다. 특히 외부 API 실패, advisor 실패, 결과 설명 실패 같은 경우가 경고 수준으로 남기 때문에, 부분 실패가 전체 장애가 되지 않으면서도 추후 분석이 가능하다. 서비스형 시스템에서 중요한 “조용한 실패를 기록하는 능력”이 있다.

### 15.7 보안 유틸리티 기능
`security.py`는 현재 프로젝트에서 보안 관련 확장 포인트 역할을 한다. 이 기능이 전체 보안 체계를 완성하는 것은 아니지만, 최소한 보안 로직이 들어갈 위치와 책임 경계를 분리해 두고 있다는 점이 중요하다. 향후 인증, 요청 제한, 민감정보 처리 강화 같은 기능을 붙이기 쉬운 구조적 자리다.

## 16. 테스트와 검증 기능

### 16.1 단위 테스트 기반 회귀 보호 기능
현재 저장소는 planner, resolver, compute API, chat API, advisor, result explanation, contract normalization 등을 폭넓게 검증하는 단위 테스트를 가지고 있다. 이 기능은 코드 그 자체가 아니라 저장소의 품질 자산이다. 리팩터링이나 기능 확장 과정에서 핵심 계약이 깨졌는지 빠르게 확인할 수 있게 해 준다.

### 16.2 통합 테스트 기반 라우트 검증 기능
`tests/v3/integration`은 실제 FastAPI 라우트를 조립해 health, job submission, chat contract 같은 표면을 검증한다. 이 기능은 함수 단위 테스트만으로는 잡히지 않는 라우팅, 직렬화, 응답 구조 문제를 잡아낸다. 웹 서버가 외부 소비자에게 어떤 계약을 제공하는지 확인하는 중요한 보호막이다.

### 16.3 실서버 HTTP smoke 테스트 기능
`test_web_server_smoke.py`는 실제 `uvicorn` 서버를 올리고 TCP로 접속해 `/`, `/static/chat.js`, `/api/chat` 같은 경로를 점검한다. 이 기능은 ASGI 내부 테스트보다 한 단계 더 현실적인 검증이다. 즉 “앱 객체는 맞다” 수준을 넘어서 “실제로 서버로 띄워도 웹 표면이 살아 있다”는 점을 확인해 준다.

### 16.4 라이브 외부 API smoke 테스트 기능
`test_live_smoke.py`는 환경변수가 설정된 경우에만 Gemini와 MolChat 실제 API를 때려보는 smoke 테스트를 제공한다. 이 기능은 기본 테스트를 외부 의존성 없이 유지하면서도, 운영 직전에는 실제 외부 통합 상태를 점검할 수 있게 해 준다. 안정성과 현실성을 둘 다 챙긴 구조라고 볼 수 있다.

### 16.5 결과 계약 검증 기능
`test_result_contract.py`는 SCF history, 에너지 단위 필드, explanation 구조 등 결과 payload가 기대한 계약을 지키는지 확인한다. 이 기능은 프론트엔드와 API 소비자 관점에서 매우 중요하다. 계산값이 맞는 것만큼이나, 그 값이 약속된 형태로 전달되는 것도 서비스 품질의 일부이기 때문이다.

### 16.6 한국어/오타 입력 검증 기능
planner와 구조 추출 테스트는 한국어 분자명, 오타, 혼합 입력이 실제로 기대한 plan으로 정규화되는지 검증한다. 이 기능은 이 프로젝트의 차별점인 자연어 UX를 지키는 품질 장치다. 사용자에게는 사소한 입력 관용성처럼 보이지만, 실제로는 제품 정체성을 지키는 핵심 테스트다.

## 17. 이 프로젝트가 종합적으로 제공하는 사용자 경험

### 17.1 “입력-계산-시각화-설명” 일체형 경험
이 프로젝트의 최종 기능은 개별 계산 함수들의 집합이 아니라, 사용자가 입력한 말 한 문장을 구조 해석, 계산, 시각화, 설명, 조언으로 연결하는 일체형 경험 자체다. 각 기능이 따로 존재하는 것이 아니라 서로 이어져 있기 때문에, 사용자는 전문 계산 패키지의 수많은 세부 옵션을 직접 몰라도 어느 정도 유도된 흐름 속에서 결과를 얻을 수 있다. 이 점이 바로 이 프로젝트를 단순 백엔드 라이브러리와 구분하는 가장 큰 기능적 특징이다.

### 17.2 연구 보조형 AI 도구로서의 성격
결국 이 프로젝트는 단순 계산 서버보다 “연구 보조형 AI 도구”에 더 가깝다. 사용자의 입력을 해석하고, 구조를 찾고, 계산을 실행하고, 결과를 설명하고, 다음 행동과 방법론까지 추천하는 흐름은 연구 작업의 여러 단계를 잇는 성격을 가진다. 따라서 이 저장소의 개별 기능들은 모두 독립된 유틸리티이면서도, 함께 묶였을 때 하나의 지능형 양자화학 작업환경이라는 더 큰 가치를 만든다.
