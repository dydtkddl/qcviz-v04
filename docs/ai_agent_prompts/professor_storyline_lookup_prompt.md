아래 프롬프트는 다른 LLM에게 그대로 전달해서 사용할 수 있도록 작성한 것이다.

```text
당신은 연구기획, 계산화학, 과학 커뮤니케이션, 교수님 보고용 발표 구조 설계에 강한 시니어 전략가다.
지금부터 첨부된 프로젝트 설명 문서를 바탕으로,
"교수님께 이 프로젝트의 의미를 설득력 있게 설명하기 위한 스토리라인"
을 설계하라.

중요:
- 첨부된 문서는 이 프로젝트의 현재 기능과 구조를 정성적으로 설명한 내부 기준 문서다.
- 당신은 이 문서를 1차 근거로 사용해야 한다.
- 이 프로젝트의 의미를 단순 기능 나열이 아니라,
  "왜 이런 프로젝트가 필요한가" -> "기존 현실의 병목은 무엇인가" -> "왜 이 구조가 의미 있는가" -> "그래서 연구실/산업/교육에서 어떤 가치가 있는가"
  흐름으로 설계해야 한다.
- 결과물은 교수님께 설명하기 위한 것이므로, 기술 자랑보다도 "문제 정의", "현실 맥락", "연구적/실용적 가치", "확장 가능성"이 중요하다.
- 절대 마케팅 문구처럼 쓰지 말고, 연구실 발표/랩미팅/지도교수 설명에 맞는 수준으로 정리하라.
- 지나치게 가벼운 표현, 스타트업식 과장, 근거 없는 미래 예측은 피하라.
- 프로젝트의 실제 기능과 연결되지 않는 허구의 가치 제안은 쓰지 말 것.
- 첨부 문서에 없는 내용은 일반적 배경지식으로 보완할 수 있지만, 그 경우 "이 프로젝트 코드로 직접 확인된 사실"과 "합리적 해석"을 구분하라.

당신의 핵심 목표:
이 프로젝트를 교수님께 설명할 때,
"이건 그냥 AI 붙인 데모가 아니라, 계산화학 실무의 병목을 줄이고 실험자-계산자 간의 간극을 줄이는 실제 인터페이스 실험이다"
라는 인식이 자연스럽게 생기도록 스토리라인을 짜는 것이다.

반드시 포함해야 하는 핵심 스토리 축:

1. MCP가 무엇인가
- MCP(Model Context Protocol)가 정확히 무엇인지
- 왜 단순 웹앱과 달리 MCP 인터페이스가 의미 있는지
- 왜 "AI 에이전트가 호출 가능한 계산화학 도구"라는 점이 중요한지
- 이 프로젝트에서 MCP는 단순 부가 기능인지, 아니면 향후 확장 관점에서 중요한 인터페이스인지

2. 양자화학 계산의 현실
- 실제 양자화학 계산은 왜 일반 사용자나 실험자에게 장벽이 높은지
- 구조 준비, 입력 파일 작성, method/basis 선택, charge/multiplicity 설정, 결과 해석이 왜 어려운지
- 단순히 계산 프로그램이 있다고 바로 쓸 수 없는 이유
- 계산화학 툴이 이미 많아도 여전히 사용 병목이 존재하는 이유

3. 계산화학자와 실험자의 간극
- 계산화학자는 계산 세팅과 결과 해석에 익숙하지만, 실험자는 그렇지 않은 경우가 많다는 점
- 실험자는 계산을 "하고 싶은 것"과 "실제로 입력 가능한 것" 사이에 큰 간극을 느낀다는 점
- 실험자 입장에서는 분자명은 알지만 method/basis/SCF/오비탈/ESP 해석은 낯선 경우가 많다는 점
- 그래서 계산 자체보다 "입력 인터페이스"와 "결과 해설"이 병목이 되는 현실

4. 실험자의 계산 병목
- 실제로 어떤 병목들이 있는지
  - 구조를 어디서 구하는지 모름
  - 어떤 계산을 해야 할지 모름
  - 전하/다중도/함수형/기저를 모름
  - 결과 숫자를 받아도 해석이 어려움
  - 계산이 실패하면 어디서부터 고쳐야 할지 모름
- 이 프로젝트가 각각의 병목에 어떻게 대응하는지

5. 왜 이 프로젝트가 필요한가
- 기존 계산엔진 자체를 대체하려는 게 아니라,
  계산엔진과 사용자 사이의 인터페이스 병목을 줄이는 시스템이라는 점
- Gemini/LLM은 계산값을 만드는 존재가 아니라 orchestration layer라는 점
- MolChat은 구조 해석의 source-of-truth 역할을 하고,
  LLM은 그 앞뒤 문맥을 정리하고 질문을 줄이는 역할이라는 점
- 그래서 이 프로젝트는 "AI가 계산을 대신한다"가 아니라
  "AI가 계산을 실제로 쓰이게 만든다"는 방향이라는 점

6. 이 프로젝트의 구조적 의미
- 자연어 입력 -> 구조 해석 -> 동적 질문 -> 계산 실행 -> 결과 설명 -> advisor
  라는 흐름이 왜 중요한지
- 고정 폼이 아니라 필요한 정보만 묻는 점이 왜 의미 있는지
- 결과를 수치로 끝내지 않고 설명과 다음 액션 추천으로 연결하는 점의 의미
- 웹앱 + MCP라는 이중 인터페이스가 갖는 연구적 의미

7. 교수님 관점에서 의미 있는 포인트
- 이 프로젝트가 단순한 앱 개발이 아니라 어떤 연구 질문을 품고 있는지
- Human-AI interaction 관점
- scientific workflow design 관점
- computational chemistry accessibility 관점
- laboratory productivity 관점
- 교육 도구로서의 잠재력
- 향후 에이전트형 연구 보조 시스템으로의 확장성

8. 발표/설명 시 피해야 할 함정
- "LLM이 계산을 잘한다" 식의 오해
- "기존 Gaussian/ORCA/PySCF를 대체한다"는 식의 과장
- "무조건 자동화"가 답이라는 단순한 주장
- 실험자들이 계산 세팅을 전혀 몰라도 된다는 식의 과도한 주장
- 기술 자랑만 많고 문제 정의가 빈약한 설명

당신이 해야 할 일:

1. 첨부 문서를 바탕으로 이 프로젝트의 실제 기능과 구조를 이해하라.
2. 교수님께 설명할 때 가장 설득력 있는 논리 흐름을 설계하라.
3. 그 흐름을 발표용 스토리라인으로 풀어라.
4. 그 스토리라인을 "룩업테이블" 형태로 정리하라.

  - 왜 중요한가

  - 왜 구조 해석(MolChat)과 LLM을 분리한 아키텍처가 중요한가
  - 왜 동적 clarification이 중요한가
  - 왜 결과 해설 기능이 중요한가
  - 왜 advisor 기능이 중요한가
  - 왜 웹앱 + MCP 이중 구조가 의미 있는가
  - 이 프로젝트의 연구실/교수님 관점 가치
  - 향후 연구 확장 가능성

출력 4. 교수님 설명용 확장 룩업테이블
- 이번에는 좀 더 전략적인 관점으로 별도 표를 하나 더 작성하라.
- 컬럼:
  - 관점
  - 교수님이 중요하게 볼 이유
  - 이 프로젝트가 제시하는 해법
  - 연구적 의미
  - 실용적 의미
  - 다음 단계 연구 질문

  - scientific interface design
  - human-in-the-loop AI
  - computational chemistry accessibility
  - research workflow automation
  - education / training
  - agentic science tooling

출력 5. 발표용 버전 3종

출력 6. 교수님이 질문할 수 있는 예상 질문과 답변

- 예시 질문 유형:
  - "기존 계산 패키지와 뭐가 다른가?"

  - "이 프로젝트가 연구인지 개발인지?"

- "기능 목록" 중심이 아니라 "문제-해법-구조-의미" 중심으로 짜라.

출력 8. 마지막 정리
- 이 프로젝트를 교수님께 설명할 때
  "무엇을 가장 강조해야 하는지"
  "무엇은 절대 과장하면 안 되는지"
  를 각각 5개 내외로 정리하라.

문체 요구:

추가 강한 요구:
```

이 프롬프트의 핵심 의도:

- 첨부된 전체 기능 설명서를 근거 문서로 활용하게 만들고
- 교수님 설명용으로 필요한 큰 서사 구조를 뽑아내고
- 단순 기능 소개가 아니라 "왜 이 프로젝트가 연구실 맥락에서 의미가 있는가"를 중심으로 정리하게 만드는 것이다.

이 프롬프트가 특히 잘 쓰이는 상황:

- 지도교수 미팅 전 스토리라인 정리
- 발표자료 초안 만들기 전 논리 구조 설계
- 제안서/연구계획서/랩미팅용 설명 문장 뽑기
- "기술 구현"이 아니라 "연구적 의미"를 앞세워 설명해야 할 때
