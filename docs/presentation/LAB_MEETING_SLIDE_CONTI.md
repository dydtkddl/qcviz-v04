📝 슬라이드 내용:
핵심 문장: 자연어 한 문장이 계산 요청서가 되고, 계산 결과는 해석 가능한 시각 자료로 돌아온다.

시스템은 분자를 해석하고, 필요한 계산 종류를 선택하고, 실행 결과를 정리한다.
브라우저 안에서 대화, 진행 상황, 결과 시각화를 하나의 흐름으로 제공한다.
즉, 계산 준비와 후처리의 상당 부분을 자동화한다.
이 프로젝트의 가치는 계산 자체보다도 계산 접근성의 재설계에 있다.
특히 실험 화학자가 질문 중심으로 계산을 활용하도록 돕는다는 점이 중요하다.
결과적으로 계산 의뢰의 병목과 해석 장벽을 동시에 낮추는 방향을 제시한다.

📊 SLIDE 4 — 양자화학 계산, 왜 필요한가?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎯 핵심 메시지:
양자화학 계산은 실험 결과를 설명하고 예측하는 보이지 않는 전자 구조 정보를 제공한다.

📝 슬라이드 내용:
실험만으로는 직접 보기 어려운 전자 구조를 계산이 보완해 준다.
HOMO와 LUMO는 분자의 반응성 방향을 이해하는 출발점이 된다.
Geometry optimization은 실제로 더 안정한 구조가 무엇인지 추정하게 해 준다.
ESP map은 전하 분포와 친핵성·친전자성 경향을 시각적으로 보여준다.
부분 전하와 dipole moment는 분자 간 상호작용을 이해하는 데 도움을 준다.
실험자가 반응성, 선택성, 결합 특성, 입체 구조를 해석할 때 계산은 강한 보조 도구가 된다.
특히 중간체를 직접 관찰하기 어려운 경우 계산은 가설 형성에 매우 유용하다.
결과적으로 계산은 실험을 대체하기보다, 실험 해석과 설계를 더 정교하게 만든다.
실험 화학에서 계산은 “왜 그런 결과가 나왔는가”를 묻는 순간 가치가 커진다.
이 프로젝트는 그 계산 도구에 접근하는 문턱을 낮추려는 시도다.

왼쪽에는 반응 플라스크나 NMR 스펙트럼, 오른쪽에는 HOMO/LUMO와 ESP 이미지를 배치한다.
가운데에 “해석 보완” 화살표를 두어 실험과 계산의 상호보완 관계를 강조한다.

연구실에서 계산을 찾는 이유는 결국 보이지 않는 정보를 보고 싶기 때문입니다.
전자 밀도나 전하 분포처럼 실험만으로 직접 얻기 어려운 것들이 여기에 포함됩니다.
예를 들어 어떤 위치가 더 반응성이 클지, 어떤 구조가 더 안정할지를 직관적으로 설명할 수 있습니다.
이 프로젝트는 계산 그 자체를 새로 발명하는 것이 아니라, 그 정보를 더 쉽게 꺼내 쓰게 만드는 도구입니다.

🎯 핵심 메시지:
기존 양자화학 요청은 사람과 소프트웨어가 여러 단계를 거치는 긴 파이프라인이다.

📝 슬라이드 내용:
실험자가 계산 아이디어를 떠올린다.
계산 화학자 또는 숙련된 사용자가 요청을 해석한다.
분자 구조를 정리하고 입력 파일을 작성한다.
계산 프로그램을 선택하고 이론 수준과 basis set을 결정한다.
Gaussian, ORCA, Multiwfn, GaussView 같은 서로 다른 도구를 오가며 계산을 수행한다.
수렴 실패나 좌표 문제를 만나면 입력을 다시 조정한다.
출력 파일에서 필요한 수치를 뽑아내고 별도 시각화 툴로 그림을 만든다.
마지막으로 결과를 해석해 실험자에게 다시 설명한다.
질문이 바뀌면 이 루프가 다시 반복된다.
즉, 계산 하나가 아니라 “요청-변환-실행-후처리-설명”의 복합 업무다.

🖼️ 비주얼 가이드:
의도적으로 복잡한 플로차트를 사용한다.
실험자 → 계산 화학자 → 입력 파일 → 계산 엔진 → 후처리 → 시각화 → 피드백 루프의 구조를 단계별 박스로 그린다.
중간에 오류와 재시도 화살표를 넣어 실제 반복성을 강조한다.

🎤 발표자 노트:
실제로 시간이 오래 걸리는 것은 계산 시간만이 아닙니다.
무엇을 계산해야 하는지 정하는 과정, 입력 파일을 만드는 과정, 결과를 사람이 다시 해석하는 과정이 다 합쳐져 있습니다.
그래서 단순한 질문 하나에도 여러 소프트웨어와 여러 사람이 개입하게 됩니다.
이 구조는 숙련자에게도 번거롭고, 비숙련자에게는 거의 진입이 불가능한 구조입니다.

문제 7: 동일한 유형의 계산 요청이 반복되는데도 자동화가 충분하지 않다.
문제 8: 결과적으로 실험자의 계산 리터러시가 쉽게 올라가지 않는다.

각 블록 아래에 실제 연구 현장에서 느끼는 문장형 불편을 짧게 배치한다.
우측 하단에는 “결과: 느린 반복 속도”를 굵게 요약한다.

🎤 발표자 노트:
저는 이 문제를 단순히 계산이 어렵다는 문장으로 끝내고 싶지 않습니다.
핵심은 전문가 의존 구조가 연구실 전체의 탐색 속도를 제한한다는 점입니다.
실험자가 바로 물어보고 바로 확인할 수 있다면 훨씬 더 많은 아이디어를 시험해 볼 수 있습니다.
반대로 지금 구조에서는 계산을 물어보는 것 자체가 큰 이벤트가 되어 버립니다.
다음으로는 왜 이 문제가 지금까지 완전히 해결되지 않았는지 보겠습니다.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎯 핵심 메시지:
계산 자동화의 필요성은 오래전부터 있었지만, 자연어 인터페이스와 실행 파이프라인의 결합은 최근에야 현실성이 높아졌다.

📝 슬라이드 내용:
기존에도 계산 워크플로를 단순화하려는 GUI나 스크립트 기반 도구는 존재했다.
하지만 대부분은 여전히 사용자가 계산 언어와 메뉴 체계를 익혀야 했다.
또한 도구가 특정 계산 엔진이나 상용 생태계에 묶여 있는 경우가 많았다.
자연어로 질문을 받고, 그 질문을 계산 절차로 번역하는 단계가 부족했다.
최근 LLM은 자유로운 문장을 구조화된 의도로 바꾸는 능력이 빠르게 향상되었다.
동시에 MCP 같은 도구 연결 방식은 LLM이 외부 계산 도구를 안정적으로 호출하게 만들었다.
오픈소스 양자화학 엔진과 웹 기반 시각화 성숙도도 함께 높아졌다.
즉, 언어 이해, 계산 실행, 결과 시각화를 하나의 사용자 경험으로 묶을 기술적 조건이 갖춰진 시점이다.
이 프로젝트는 바로 그 교차 지점에서 연구실 실사용 가능성을 시험하는 사례다.

🖼️ 비주얼 가이드:
과거에는 분리돼 있던 세 원을 그리고, 현재 시점에서 겹치는 교집합을 강조한다.
원 1: 자연어 이해, 원 2: 계산 엔진, 원 3: 웹 시각화/자동화.
교집합 중앙에 “LLM + Tool Orchestration + Open QC”를 배치한다.

🎤 발표자 노트:
중요한 점은 우리가 완전히 새로운 계산 방법을 만든 것이 아니라, 여러 성숙한 기술을 연결했다는 것입니다.
예전에는 이 연결이 불안정하거나 너무 수작업이 많아서 실사용성이 낮았습니다.
지금은 자연어를 구조화하는 능력과 외부 도구를 호출하는 방식이 훨씬 좋아졌습니다.
그래서 연구실 수준에서 실제로 써볼 수 있는 자동화 플랫폼이 가능해졌습니다.

📝 슬라이드 내용:
사용자는 “이 분자의 HOMO를 보여줘”처럼 자연어로 질문한다.
시스템은 분자명과 원하는 분석 종류를 문장에서 추출한다.
필요한 계산 종류를 자동으로 선택하고, 구조를 준비하고, 계산을 실행한다.
결과는 숫자 파일로 끝나지 않고 3D 시각화와 해석 문장으로 정리된다.
진행 상황은 실시간으로 브라우저에 전달된다.
후속 질문이 들어오면 이전 계산 맥락을 이어서 활용할 수 있다.
즉, 입력 형식의 복잡성을 질문 중심 인터페이스로 바꾸는 것이 핵심이다.
기존에는 사용자가 계산 절차를 배워야 했다면, 이제는 시스템이 절차를 대신 조직한다.

우리는 그 사이 번역기를 시스템 안에 넣은 셈입니다.
그리고 그 번역 결과가 실제 계산까지 이어지도록 만든 것이 핵심입니다.

📊 SLIDE 9 — Key Technology #1 — PySCF
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎯 핵심 메시지:
PySCF는 실제 양자화학 계산을 수행하는 오픈소스 엔진으로, 이 플랫폼의 계산적 기반이다.

📝 슬라이드 내용:
PySCF는 Python Simulations of Chemistry Framework의 약자다.
역할: 분자에 대한 전자 구조 계산을 수행하는 오픈소스 양자화학 엔진이다.
우리 프로젝트에서는 single-point energy, orbital, ESP, partial charge, geometry optimization 계산에 사용된다.
Python 생태계에 잘 연결되기 때문에 자동화 파이프라인과 결합하기 좋다.
상용 소프트웨어처럼 입력 파일 중심이 아니라 코드 기반 조합이 쉬워 자동화에 유리하다.
오픈소스라는 점은 라이선스 비용 없이 재현 가능한 연구 환경을 만든다는 의미가 있다.
프로젝트 저장소에서도 핵심 계산 모듈은 PySCF runner를 중심으로 구성되어 있다.
즉, 챗봇이 실제로 계산을 “말만” 하는 것이 아니라, PySCF를 통해 계산을 수행한다.
자동화 시스템의 신뢰성은 결국 이 계산 엔진의 안정적 실행에 달려 있다.

🖼️ 비주얼 가이드:
PySCF 로고 또는 텍스트 로고를 좌측에, 우측에 계산 기능 목록을 아이콘으로 정리한다.
기능 아이콘 예시: 에너지, 오비탈, ESP, 최적화, 전하.
하단에는 “Open-source, Python-native, automation-friendly” 세 단어를 강조한다.

🎤 발표자 노트:
여기서 중요한 것은 LLM이 계산을 대신하는 것이 아니라는 점입니다.
실제 물리 계산은 여전히 검증된 양자화학 엔진이 담당합니다.
LLM은 인터페이스와 조정자 역할을 하고, 계산 자체는 PySCF가 수행합니다.
그래서 이 프로젝트는 AI 데모가 아니라 계산 자동화 시스템이라고 보는 것이 더 정확합니다.
다음은 이 자연어 인터페이스를 가능하게 하는 LLM 역할입니다.

📊 SLIDE 10 — Key Technology #2 — LLM
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎯 핵심 메시지:
LLM은 사용자의 자유로운 문장을 계산 의도로 해석하고 결과를 이해 가능한 설명으로 바꿔주는 자연어 인터페이스다.

📝 슬라이드 내용:
LLM은 Large Language Model의 약자다.
역할: 사람의 문장을 이해하고 구조화된 작업 지시로 바꾸는 언어 모델이다.
우리 프로젝트에서는 사용자의 질문에서 분자명, 계산 종류, 후속 요청을 추출하는 데 사용된다.
예를 들어 “아세톤의 HOMO를 보여줘”에서 분자와 의도를 동시에 파악한다.
또한 결과를 다시 사람이 읽기 쉬운 문장으로 요약하고 설명하는 데도 활용된다.
질문이 모호할 경우 추가 질문을 던져 clarification을 수행할 수 있다.
프로젝트 구조상 Gemini 또는 OpenAI 기반 선택지가 있으며, 실패 시 휴리스틱 경로도 남아 있다.
즉, LLM은 계산기를 대체하는 것이 아니라 사용자의 의도를 계산 체계와 연결하는 프런트 도어다.
연구실 사용자 입장에서는 입력 포맷이 아니라 대화가 인터페이스가 된다는 점이 핵심 변화다.

🖼️ 비주얼 가이드:
사용자 문장 예시와 추출된 구조화 정보 카드 두 개를 연결해 보여준다.
예시: “benzene HOMO 보여줘” → 분자: benzene / 작업: orbital preview.
오른쪽 하단에는 “interpret → route → explain” 3단계를 배치한다.

🎤 발표자 노트:
LLM이 들어간다고 하면 종종 계산 결과 자체를 생성한다고 오해할 수 있습니다.
하지만 여기서는 계산 결과를 꾸며내는 용도가 아니라, 사용자의 언어를 작업 언어로 바꾸는 역할이 더 중요합니다.
그리고 계산이 끝난 뒤에는 결과를 다시 자연어로 설명해 주는 통역사 역할도 합니다.
이 덕분에 실험자는 계산 용어를 완벽히 몰라도 질문하고 해석할 수 있습니다.
이제 이 LLM이 실제 도구를 어떻게 움직이는지 MCP 관점에서 설명드리겠습니다.

📊 SLIDE 11 — Key Technology #3 — MCP
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎯 핵심 메시지:
MCP는 언어 모델이 외부 계산 도구를 호출하도록 연결해 주는 실행 인터페이스 개념이다.

📝 슬라이드 내용:
MCP는 Model Context Protocol의 약자다.
역할: LLM이 외부 도구와 상호작용할 때 필요한 입력과 출력을 정리해 주는 연결 규약이다.
쉽게 말해 LLM의 손과 발 같은 역할을 한다.
LLM이 “무엇을 해야 하는지”를 판단하면, MCP 스타일의 도구 호출이 “어떻게 실행할지”를 담당한다.
프로젝트 저장소에는 ESP 계산, orbital 시각화, partial charge 계산, 분석 보조 기능 등의 도구 모듈이 존재한다.
즉, 언어 이해와 계산 실행 사이에 도구 계층을 둔 구조다.
현재 사용자-facing 경로는 웹 앱 중심이지만, 내부 설계는 여전히 tool-oriented 성격을 강하게 유지한다.
이 구조 덕분에 향후 새로운 계산 기능을 모듈식으로 추가하기 쉽다.
연구 자동화에서 중요한 것은 LLM의 말하기 능력보다, 실제로 도구를 안전하게 호출하는 능력이다.

🖼️ 비주얼 가이드:
ASCII 예시를 직접 넣는다.
사용자 → LLM → MCP 도구층 → PySCF/Resolver/Visualizer 로 이어지는 4단 구조를 박스로 그린다.
각 박스 아래에 한 줄 역할 설명을 넣는다.

🎤 발표자 노트:
MCP라는 용어가 익숙하지 않더라도, 여기서는 LLM과 계산 도구를 연결하는 인터페이스라고 이해하시면 충분합니다.
핵심은 언어 모델이 혼자 답만 하는 것이 아니라, 실제 계산 도구를 불러 일하게 만든다는 점입니다.
그래서 저는 MCP를 LLM의 손과 발이라고 비유하고 싶습니다.
사용자가 문장으로 주문하면, 시스템은 도구 조합으로 그것을 실제 계산 작업으로 바꿉니다.
이제 이 세 가지가 합쳐진 전체 파이프라인을 한 장으로 보겠습니다.

플랫폼은 사용자 질문을 받아 구조 해석, 계산 실행, 결과 설명, 브라우저 시각화까지 하나의 흐름으로 연결한다.

📝 슬라이드 내용:
사용자는 웹 브라우저에서 자연어로 계산을 요청한다.
대화 인터페이스는 질문을 분석해 분자명과 작업 유형을 추출한다.
필요하면 외부 분자 정보 소스에서 구조를 찾거나 보정한다.
작업 종류에 따라 PySCF 계산 파이프라인이 선택된다.
계산 중 진행 상황은 실시간으로 브라우저에 스트리밍된다.
계산 결과는 에너지, 전하, 오비탈, ESP, 최적화 구조 등으로 정리된다.
시각화 레이어가 cube 기반 3D 렌더링과 결과 패널을 구성한다.
추가 설명 모듈이 결과 해석과 추천 정보를 텍스트로 정리한다.
최종적으로 사용자는 계산 파일이 아니라 이해 가능한 결과 화면을 받는다.

🖼️ 비주얼 가이드:
ASCII 블록 다이어그램 제안:
사용자 → 웹 챗봇 → 의도 분석/도구 호출 → 구조 해석 → PySCF 계산 → 결과 정리/시각화 → 사용자
보조 흐름으로 Redis/worker는 점선 옵션 경로로 표시한다.
브라우저와 백엔드 사이에는 WebSocket 진행 상황 스트림을 별도 화살표로 표시한다.

📊 SLIDE 13 — 파이프라인 — Step by Step
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎯 핵심 메시지:
시스템은 질문을 받아 7단계 정도의 연속 절차로 계산 결과를 만든다.

📝 슬라이드 내용:
Step 1. 사용자 입력: 분자와 원하는 분석을 자연어로 제시한다.
Step 2. 의도 분석: 시스템이 분자명, 계산 종류, 후속 맥락을 파악한다.
Step 3. 구조 준비: 분자 구조를 조회하거나 재구성하고 필요한 경우 전하/조성을 확인한다.
Step 4. 계산 선택: orbital, ESP, partial charge, optimization 등 적절한 작업을 고른다.
Step 5. 계산 실행: PySCF 엔진이 실제 전자 구조 계산을 수행한다.
Step 6. 결과 정리: 수치, cube, 좌표, 설명용 텍스트를 묶는다.
Step 7. 시각화 및 응답: 브라우저에 3D 결과와 해석 문장을 보여준다.
필요 시 Step 8. 후속 질문: “이번엔 LUMO도 보여줘” 같은 연속 대화를 이어간다.
즉, 계산 요청이 단발성 명령이 아니라 대화형 세션으로 발전할 수 있다.

🖼️ 비주얼 가이드:
가로형 타임라인을 사용한다.
각 단계마다 번호 원형 배지와 짧은 키워드를 붙인다.
Step 5와 Step 7은 색을 달리해 “계산”과 “사용자 경험”을 구분한다.

🎤 발표자 노트:
이 순서도를 보면 시스템이 단순히 API 하나를 감싼 것이 아니라는 점이 보입니다.
실제로는 구조 준비, 계산 선택, 결과 정리 같은 중간 조정 단계가 중요합니다.
특히 실험자 관점에서는 1단계와 7단계만 보이지만, 그 사이에 많은 자동화가 들어갑니다.
이런 중간 단계가 잘 설계되어야 결과 품질과 사용자 경험이 동시에 좋아집니다.
조금 더 직관적으로 이해할 수 있도록 일상적인 비유로 다시 설명해 보겠습니다.

📊 SLIDE 16 — Feature Deep Dive #1 — HOMO/LUMO 계산
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎯 핵심 메시지:
HOMO/LUMO 시각화는 분자의 반응성과 전자 이동 가능성을 직관적으로 보여주는 대표 기능이다.

📝 슬라이드 내용:
프론티어 오비탈 이론은 반응성이 높은 전자 궤도에 주목하는 관점이다.
HOMO는 전자를 내놓기 쉬운 방향을, LUMO는 전자를 받기 쉬운 방향을 시사한다.
사용자는 분자명과 함께 HOMO 또는 LUMO를 보여 달라고 요청할 수 있다.
시스템은 orbital preview 작업을 선택하고 필요한 전자 구조 계산을 수행한다.
결과는 cube 기반 데이터와 3D 등치면 시각화로 제공된다.
사용자는 브라우저에서 회전, 확대, 다른 orbital 선택 같은 상호작용을 할 수 있다.
화학적으로는 반응성 예측, 친핵/친전자 공격 위치 가설 설정에 유용하다.
교육적으로도 전자 구조 개념을 직관적으로 학습시키는 데 도움이 된다.
연구실 데모에서는 가장 먼저 시각적 임팩트를 주기 좋은 기능이다.

🖼️ 비주얼 가이드:
실제 HOMO/LUMO 렌더링 예시 이미지를 전면에 배치한다.
오른쪽에는 “사용자 질문 → 계산 → 3D 결과”의 3단 소형 플로를 붙인다.
가능하다면 같은 분자에 대해 HOMO와 LUMO를 나란히 비교하는 레이아웃을 사용한다.

🎤 발표자 노트:
이 기능은 발표에서 청중의 관심을 가장 쉽게 끌 수 있는 기능입니다.
오비탈은 계산 화학의 대표적인 시각 언어이기 때문에, 시스템 가치가 빠르게 전달됩니다.
실험자에게는 반응성에 대한 직관을 제공하고, 학생에게는 개념 교육 도구 역할도 합니다.
사용자는 복잡한 cube 처리 과정을 전혀 몰라도 결과를 바로 볼 수 있습니다.
다음은 구조 자체를 안정화시키는 geometry optimization 기능입니다.

📊 SLIDE 17 — Feature Deep Dive #2 — Geometry Optimization
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎯 핵심 메시지:
Geometry optimization은 더 안정한 분자 구조를 찾는 기본 단계이며, 후속 분석의 출발점이 된다.

📝 슬라이드 내용:
실험에서 떠올린 초기 구조는 반드시 에너지 최소 구조라고 볼 수 없다.
Geometry optimization은 주어진 이론 수준에서 더 안정한 좌표를 찾는 계산이다.
프로젝트는 geometry_optimization 작업을 별도 지원한다.
사용자는 자연어로 구조 최적화를 요청할 수 있다.
최적화가 끝나면 갱신된 3D 좌표와 구조 정보를 브라우저에 보여준다.
이 단계는 이후 orbital, ESP, charge 분석의 기반 구조를 정리하는 역할을 한다.
실험자 관점에서는 “이 구조가 말이 되는가”를 빠르게 확인하는 도구가 된다.
계산적 세부 이론 수준은 시스템이 정한 설정과 추천 흐름에 따라 적용된다.
따라서 구조 최적화는 단독 기능이자 다른 기능들의 전처리 성격도 갖는다.

🖼️ 비주얼 가이드:
최적화 전후 구조를 좌우 비교하는 이미지를 배치한다.
중간에 “energy decrease” 화살표를 넣는다.
아래에는 최적화 결과가 후속 분석으로 이어지는 간단한 분기 화살표를 둔다.

🎤 발표자 노트:
많은 계산 결과는 어떤 구조를 기준으로 하느냐에 따라 해석이 달라질 수 있습니다.
그래서 구조 최적화는 단순 기능 하나가 아니라 다른 계산의 출발점이라는 의미가 큽니다.
실험자 입장에서는 복잡한 입력 파일 없이 구조 안정화를 요청할 수 있다는 점이 중요합니다.
이후 오비탈이나 ESP를 볼 때도 더 일관된 기준 구조를 확보할 수 있습니다.
다음으로는 전하 분포를 색으로 보여 주는 ESP 기능을 보겠습니다.

📊 SLIDE 18 — Feature Deep Dive #3 — ESP Mapping
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎯 핵심 메시지:
ESP 시각화는 분자 표면 위의 전하 분포를 색으로 표현해 상호작용 경향을 직관적으로 보여준다.

📝 슬라이드 내용:
ESP는 Electrostatic Potential의 약자다.
분자 주변에서 전하가 어떻게 느껴지는지를 시각화하는 방법으로 이해할 수 있다.
프로젝트는 esp_map 작업을 통해 ESP 계산과 시각화를 지원한다.
결과는 보통 빨강과 파랑 계열 색상으로 전자 풍부 영역과 상대적 전자 부족 영역을 보여준다.
실험자는 이 지도를 통해 친핵성 또는 친전자성 상호작용 위치를 직관적으로 가늠할 수 있다.
오비탈 그림보다 더 표면적이고 직관적인 그림을 원하는 경우 특히 유용하다.
브라우저에서 3D로 회전하며 분자 표면을 확인할 수 있다.
프로젝트 README와 웹 시각화 구성에서 ESP surface는 핵심 데모 기능으로 다뤄진다.
즉, ESP는 설명력과 시각적 전달력이 모두 높은 대표 기능이다.

📝 슬라이드 내용:
시각화는 브라우저 기반 3D 분자 뷰어를 통해 제공된다.
오비탈과 ESP는 cube 기반 표면 데이터로 렌더링된다.
결과 패널은 계산 요약과 수치 정보를 함께 보여준다.
사용자는 분자를 회전하고 확대하며 관심 각도에서 볼 수 있다.
HOMO와 LUMO 같은 서로 다른 orbital을 전환하는 상호작용이 가능하다.
ESP surface는 표면 색 분포 중심으로 보여 줄 수 있다.
최적화 결과 구조도 동일한 화면 안에서 이어서 확인할 수 있다.
이런 통합 시각화는 별도 데스크톱 프로그램 의존도를 줄인다.
즉, 계산 결과 전달의 마지막 마일을 담당하는 핵심 구성 요소다.

| 반복 질의 | 사람 간 재의뢰 필요 | 세션 continuation 가능 |
| 비용 구조 | 상용 도구 의존 가능성 | 오픈소스 기반 중심 |
정확한 시간 절감 수치는 향후 사용자 테스트로 계량화하는 것이 바람직하다.
그러나 구조적 비교만으로도 병목 감소 방향은 분명하다.

🖼️ 비주얼 가이드:

시간 절감은 정량 숫자 대신 “days to minutes-scale interaction” 같은 개념 표현으로 과장 없이 표시한다.
색상은 기존 방식은 회색, 새 방식은 청록으로 대비시킨다.

향후 사용자 평가와 benchmark가 보강되면 학술적 설득력이 더 높아진다.
연구 질문은 “새 알고리즘을 만들었는가”가 아니라 “새로운 접근성을 만들어 냈는가”에 가깝다.

📝 슬라이드 내용:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎯 핵심 메시지:
플랫폼은 단순 요청-응답을 넘어 계산 진행 상황을 추적하고, 필요하면 작업 실행을 분리할 수 있는 구조를 갖는다.

📝 슬라이드 내용:
기본 모드는 웹 서버 내부에서 계산 작업을 실행하는 단순한 구조다.
하지만 저장소에는 Redis 기반 job store와 arq worker 경로가 이미 존재한다.
즉, 계산 작업을 별도 worker로 보내는 split-runtime 모델을 선택적으로 사용할 수 있다.
브라우저에는 WebSocket을 통해 실시간 진행 상황이 전달된다.
사용자는 단순히 기다리는 것이 아니라 현재 어떤 단계가 진행되는지 볼 수 있다.
이는 긴 계산 작업에서 사용자 이탈을 줄이는 데 중요하다.
멀티유저 환경이나 계산량 증가를 고려하면 worker 분리 구조의 의미가 더 커진다.
따라서 이 프로젝트는 프로토타입이지만 운영 확장에 대한 준비도 일부 포함하고 있다.

🖼️ 비주얼 가이드:
웹 서버와 worker를 분리한 다이어그램을 그린다.
브라우저 ↔ 웹 서버는 WebSocket, 웹 서버 ↔ worker는 queue/Redis 화살표로 표시한다.
기본 모드와 확장 모드를 두 개의 작은 서브그림으로 비교하면 좋다.

🎤 발표자 노트:
긴 계산은 사용자 경험이 쉽게 나빠질 수 있기 때문에 진행 상황 표시가 중요합니다.
이 프로젝트는 그 점을 고려해서 실시간 업데이트 구조를 갖추고 있습니다.
또한 현재는 단일 서버로도 돌아가지만, 향후 사용자가 늘면 워커 분리가 필요할 수 있습니다.
저장소에 이미 그 경로가 들어 있다는 점은 확장 논의에서 좋은 근거가 됩니다.
다음 백업 슬라이드는 계산 엔진 자체의 이론적 배경을 간단히 짚습니다.

📊 SLIDE 37 — Backup — PySCF 이론적 배경 요약
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎯 핵심 메시지:
발표에서 깊게 들어가지는 않더라도, PySCF가 전자 구조 계산 프레임워크라는 점은 배경으로 이해해 둘 필요가 있다.

📝 슬라이드 내용:
PySCF는 Hartree-Fock, DFT 등 다양한 전자 구조 계산을 수행할 수 있는 Python 기반 프레임워크다.
전자 구조 계산은 분자의 wavefunction 또는 electron density를 바탕으로 에너지와 성질을 추정한다.
HOMO/LUMO, 부분 전하, dipole moment, ESP 같은 값은 이 계산 결과에서 파생된다.
Geometry optimization은 에너지 표면에서 더 안정한 좌표를 찾는 과정이다.
이 프로젝트는 계산 이론 자체를 바꾸지 않고, 기존 이론 기반 계산을 더 접근 가능하게 만든다.
즉, scientific validity는 계산 엔진과 설정의 적절성, benchmark, 입력 구조 품질에 의해 뒷받침된다.
발표에서는 이 슬라이드를 백업용으로 두고, 질문이 나올 때만 활용하면 충분하다.
핵심은 AI가 계산을 대신하는 것이 아니라 계산을 연결하고 설명한다는 점이다.

🖼️ 비주얼 가이드:
간단한 개념도: molecule → electronic structure calculation → properties.
property examples로 energy, orbital, charge, ESP를 네 개의 박스로 놓는다.
수식은 최소화하고 개념 화살표 중심으로 정리한다.

🎤 발표자 노트:
계산 화학 전공자 질문이 나오면 이 슬라이드가 기본 방어선이 될 수 있습니다.
우리는 새로운 양자화학 이론을 제안하는 것이 아니라, 기존 이론 기반 계산에 대한 접근 인터페이스를 바꾸는 것입니다.
따라서 scientific validity의 핵심은 benchmark와 설정 선택에 있습니다.
이 점을 분명히 해야 AI 프로젝트로 오해받지 않고 계산 프로젝트로도 인정받을 수 있습니다.
다음은 MCP와 도구 계층을 조금 더 직접적으로 보여 주는 백업 슬라이드입니다.

프로젝트는 사용자 질문을 여러 계산 도구 기능으로 분해하는 tool-oriented 설계를 유지하고 있다.

📝 슬라이드 내용:
