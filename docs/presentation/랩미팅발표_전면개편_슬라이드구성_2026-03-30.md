## 개편 핵심

- 발표의 주인공은 `MCP`가 아니라 `QCViz`다.
- QCViz는 `실험자를 위한 web-first 올인원 계산화학 플랫폼`으로 정의한다.
- `LLM`은 계산기가 아니라 자연어를 계산 가능한 요청으로 번역하는 인터페이스 계층으로 설명한다.
- `MolChat`은 구조 grounding 계층으로, `PySCF`는 실제 계산 엔진으로 분리해 설명한다.
- 현재 메인 경로는 `web-first direct orchestration`이며, `MCP`는 호환 가능한 외부 도구 인터페이스로 위치시킨다.
- 발표 흐름은 `문제 -> 정의 -> 가능 조건 -> 시스템 -> 신뢰성 -> 의미` 순으로 밀어준다.

## 현재 deck 정리 방향

- 기존 21~24번 `Conclusion & Outlook`은 `연구실 임팩트`, `포지셔닝`, `Future Work`, `Summary & Discussion`으로 재편한다.
- `Thank You`는 유지한다.

## 전체 서사

1. 실험 연구자는 계산이 필요하지만, 현재 워크플로는 너무 많은 단계와 전환을 요구한다.
2. QCViz는 그 경로를 웹에서 단일화하려는 플랫폼이다.
3. 이 플랫폼의 본질은 `AI가 계산을 대신하는 것`이 아니라, `질문을 실제 계산과 시각화로 연결하는 것`이다.
4. 그 연결은 `LLM`, `MolChat`, `PySCF`, 브라우저 시각화가 역할을 나눠 수행한다.
5. 따라서 이 시스템은 계산값을 꾸며내는 챗봇이 아니라, 실제 계산에 접근하는 경로를 재설계한 플랫폼이다.
6. 이 플랫폼이 자리 잡으면 연구실에서 계산을 사용하는 방식 자체가 바뀔 수 있다.

## 슬라이드 구조

### Slide 1. 실험 질문과 전자구조 계산 사이의 간극
핵심 메시지:
- 발표의 출발점은 새 이론이 아니라 접근 경로의 문제다.
- 실험자의 질문은 구체적이지만, 계산으로 가는 경로는 여전히 길고 파편화돼 있다.

슬라이드용 문구:
- Experimental question
- fragmented computational workflow
- web-first access to real quantum chemistry

권장 시각 요소:
- 실험 질문에서 복잡한 도구 체인으로 빠지는 기존 흐름 vs 단일 웹 흐름 비교

### Slide 2. 오늘 답할 세 가지 질문
핵심 메시지:
- 왜 이런 플랫폼이 필요한가
- 정확히 무엇을 만들었는가
- 왜 믿을 수 있고, 왜 의미가 있는가

슬라이드용 문구:
- 왜 필요한가
- 어떻게 작동하는가
- 무엇이 바뀌는가

### Slide 3. 왜 실험 연구에 계산이 필요한가
핵심 메시지:
- HOMO/LUMO, ESP, 최적화는 논문용 그림이 아니라 해석 도구다.
- 실험자는 거대한 프로젝트보다 즉각적인 질문에 답을 원한다.

슬라이드용 문구:
- 반응성 해석
- 선택성 설명
- 구조 안정성 확인
- 상호작용 예측

### Slide 4. 현재 워크플로가 왜 느리고 불편한가
핵심 메시지:
- 병목은 계산 엔진보다도 구조 준비, 입력 작성, 후처리, 시각화 전환에 있다.
- 이 구조는 계산을 소수 전문가에게 집중시키고, 실험자는 기다리게 만든다.

슬라이드용 문구:
- structure preparation
- input building
- engine execution
- parsing and visualization
- repeated handoff

### Slide 5. 우리가 바꾸려는 것은 무엇인가
핵심 메시지:
- 질문에서 구조 확인, 계산, 시각화까지 이어지는 경로를 하나의 사용자 경험으로 바꾸는 것이다.
- 목표는 자동화가 아니라 계산 접근의 재설계다.

슬라이드용 문구:
- Question -> structure -> compute -> visualize
- One browser workflow
- Lower barrier, higher frequency

### Slide 6. QCViz는 무엇이고 무엇이 아닌가
핵심 메시지:
- QCViz는 실험자를 위한 web-first 플랫폼이다.
- 계산값을 지어내는 AI 시스템이 아니고, 현재 전체 경로가 MCP-native인 시스템도 아니다.

슬라이드용 문구:
- It is:
- web-first conversational computational chemistry platform
- It is not:
- a result-fabricating chatbot
- a fully MCP-native product

### Slide 7. 사용자 경험: 질문에서 시각화까지 한 흐름
핵심 메시지:
- 사용자는 자연어로 질문하고, 필요하면 구조를 확인하고, 계산 결과를 같은 화면에서 바로 본다.
- 복잡한 시각화 툴체인을 따로 열지 않아도 된다.

슬라이드용 문구:
- Natural-language query
- structure confirmation
- reasonable defaults
- browser-native visualization
- follow-up conversation

### Slide 8. 왜 지금 가능한가
핵심 메시지:
- LLM, MolChat, PySCF, 웹 인터페이스가 모두 실용 수준에 도달했기 때문에 지금 이 구성이 가능해졌다.

슬라이드용 문구:
- LLM for intent translation
- MolChat for grounding
- PySCF for real DFT
- web UI for integrated delivery

### Slide 9. 시스템 전체 구조
핵심 메시지:
- QCViz는 하나의 앱처럼 보이지만 내부는 역할이 분리돼 있다.
- 메인 사용자 경로는 web-first direct orchestration이다.

슬라이드용 문구:
- Web interface
- LLM planning layer
- MolChat / structure resolution
- compute orchestration
- PySCF backend
- result packaging and visualization

권장 시각 요소:
- 좌에서 우로 이어지는 아키텍처 다이어그램

### Slide 10. LLM의 역할: 계산이 아니라 번역과 라우팅
핵심 메시지:
- LLM은 분자, 작업 종류, 문맥, clarification 필요 여부를 읽는다.
- 최종 과학적 결과를 생성하지는 않는다.

슬라이드용 문구:
- intent parsing
- task routing
- clarification
- follow-up context
- no fabricated energy values

### Slide 11. 구조 grounding: MolChat과 clarification의 역할
핵심 메시지:
- 사용자의 구조 표현은 불완전하거나 애매할 수 있다.
- 그래서 구조를 먼저 grounding하고, 필요하면 확인을 받은 뒤 계산으로 넘어간다.

슬라이드용 문구:
- molecule grounding
- ambiguity handling
- clarification before compute
- structure lock

### Slide 12. 계산 실행: direct orchestration과 PySCF
핵심 메시지:
- 현재 웹 메인 경로는 내부 direct orchestration으로 PySCF를 호출한다.
- 이 선택은 성능, 실패율, 사용자 경험 기준에서 합리적이다.
- MCP는 외부 연동을 위한 호환 계층으로 남긴다.

슬라이드용 문구:
- web-first direct orchestration
- PySCF executes the calculation
- MCP-compatible, not MCP-native end-to-end

### Slide 13. 결과 패키징과 브라우저 시각화
핵심 메시지:
- 계산이 끝난 뒤 결과를 파일로 던져놓는 것이 아니라, 바로 읽고 비교할 수 있는 화면으로 돌려준다.
- 이 지점이 기존 워크플로와 사용자 체감 차이를 만든다.

슬라이드용 문구:
- orbital
- ESP
- geometry result
- summary and follow-up
- no separate visualization toolchain

### Slide 14. 사례 1: 단일 분자 HOMO
핵심 메시지:
- 하나의 간단한 자연어 요청이 실제 계산과 시각화까지 연결되는 흐름을 보여준다.

슬라이드용 문구:

### Slide 15. 사례 2: follow-up과 연속 질의
핵심 메시지:

슬라이드용 문구:

### Slide 17. 정확도와 신뢰성 근거
핵심 메시지:
- QCViz의 신뢰성은 `LLM이 똑똑하다`가 아니라 `PySCF/GPU4PySCF가 기존 코드들과 교차검증되어 있다`는 데 있다.
- ORCA 직접 전용 비교가 약하다는 점도 같이 밝힌다.

슬라이드용 문구:
- Q-Chem 6.1: single-digit μHartree agreement
- ORCA/LSDalton: μHartree-level agreement via GPU4PySCF
- SIESTA: chemical accuracy maintained
- GitHub #688: grid sensitivity example

권장 시각 요소:

### Slide 18. 성능과 현실적 한계
핵심 메시지:
- PySCF CPU가 항상 가장 빠른 것은 아니다.
- 그러나 GPU4PySCF와 설정 최적화가 들어가면 상황이 크게 달라질 수 있다.
- 이 플랫폼의 가치는 절대 속도만이 아니라 접근 경로 단일화에도 있다.

슬라이드용 문구:
- CPU PySCF is not always fastest
- GPU4PySCF can be dramatically faster
- speed is one criterion, not the whole story
- current limitations remain
