# QCViz-MCP v3 Master Specification and Contribution Narrative

## 1. 프로젝트 개요 / Project Overview

QCViz-MCP v3는 실험 화학자와 계산 화학자 사이에 존재하는 실제 연구 워크플로우의 간극을 줄이기 위해 설계된 웹 기반 대화형 양자화학 계산 플랫폼이다. 이 프로젝트의 핵심 목표는 사용자가 자연어로 분자와 계산 의도를 입력하면, 시스템이 그 입력을 계산 가능한 구조 질의로 정규화하고, 필요한 경우 의미 기반 후보 확인 과정을 거쳐, 실제 양자화학 엔진을 실행하고, 최종적으로 결과를 브라우저에서 바로 해석 가능한 형태로 돌려주는 것이다. 다시 말해 이 플랫폼은 단순한 챗봇이 아니라, 자연어 입력, 구조 해석, 계산 실행, 결과 설명, 시각화, 후속 질문 처리까지를 하나의 일관된 사용자 경험 안에 연결하는 연구용 시스템이다.

QCViz-MCP v3 is a browser-based conversational quantum chemistry platform designed to reduce the real workflow gap between experimental chemists and computational chemistry practitioners. The central goal of the project is to let a user express a molecule and an analysis intent in natural language, have the system normalize that request into a computation-ready structure query, perform semantic grounding and clarification when necessary, run a real quantum chemistry engine, and finally return outputs in a browser-native, interpretable form. In other words, the platform is not merely a chatbot. It is a research system that integrates natural-language intake, structure resolution, computation, explanation, visualization, and follow-up interaction inside one coherent user experience.

이 프로젝트는 “LLM이 계산을 대신한다”는 서사를 따르지 않는다. 오히려 LLM과 규칙 기반 정규화, MolChat 기반 구조 grounding, PySCF 계산 경로, 웹소켓 기반 상태 전달, 브라우저 시각화 계층을 명확히 분리하여, 어떤 단계가 사용자 질의를 해석하는 단계인지, 어떤 단계가 실제 계산을 수행하는 단계인지, 어떤 단계가 결과를 사람이 읽을 수 있게 재구성하는 단계인지를 구분한다. 이러한 아키텍처적 분리는 신뢰성과 설명 가능성을 동시에 확보하기 위한 설계 선택이다.

The project does not follow the simplistic narrative that “the LLM performs chemistry by itself.” Instead, it explicitly separates LLM-supported intent interpretation, rule-assisted normalization, MolChat-based structure grounding, PySCF-backed computation, websocket-driven state delivery, and browser visualization. This separation makes it possible to distinguish between the stage that interprets a user request, the stage that performs the actual scientific computation, and the stage that repackages the result into a human-readable form. This architectural separation is a deliberate design choice made in order to support both reliability and interpretability.

## 2. 문제 정의와 연구 배경 / Problem Definition and Research Background

실험 화학 연구 환경에서 계산은 더 이상 선택적 부가물이 아니라, 반응성 해석, 구조 안정성 검토, 전자 구조 해석, 선택성 논의, 메커니즘 가설 검증을 위한 핵심 도구가 되었다. 그러나 현실의 워크플로우는 여전히 분절되어 있다. 사용자는 분자 구조를 준비하고, 적절한 계산 조건을 정하고, 계산 프로그램을 실행하고, 출력 파일을 해석하고, 별도 시각화 도구로 오비탈 또는 ESP를 렌더링해야 한다. 이 과정은 계산 자체보다 전처리와 후처리, 파일 포맷 이동, 프로그램 간 왕복, 숙련자 의존도가 더 큰 병목이 되는 경우가 많다.

In experimental chemistry settings, computation is no longer an optional add-on. It has become a core instrument for interpreting reactivity, checking structural stability, understanding electronic structure, discussing selectivity, and testing mechanistic hypotheses. Yet the real workflow remains fragmented. A user must prepare a molecular structure, choose a computational setup, run a chemistry package, interpret the output files, and often use separate visualization software to render orbitals or electrostatic potentials. In practice, the strongest bottleneck is often not the calculation itself but the pre-processing, post-processing, file-format transitions, tool switching, and dependence on expert users.

이 프로젝트가 겨냥하는 문제는 정확히 이 지점에 있다. 연구실의 많은 질의는 거대한 고성능 계산 캠페인에서 출발하지 않는다. 오히려 “이 분자의 HOMO를 보고 싶다”, “ESP 분포를 보고 싶다”, “이 구조를 최적화해 보고 싶다”, “같은 분자에 대해 후속 분석을 이어서 해 보고 싶다”와 같이 작고 구체적이며 맥락적인 질문에서 시작한다. 기존의 도구 체인은 이런 질문에 지나치게 무겁고, 챗봇형 인터페이스는 반대로 계산의 엄밀성을 보장하지 못하는 경우가 많다. QCViz-MCP v3는 이 두 극단 사이에서, 실제 계산 엔진과 실제 사용자 인터페이스를 연결하는 실용적 중간 지대를 만드는 것을 목표로 한다.

This project targets precisely that pain point. Many laboratory questions do not begin as large-scale, high-performance campaigns. They begin as small, concrete, contextual requests such as “show me the HOMO of this molecule,” “map the ESP,” “optimize this structure,” or “continue with a follow-up analysis on the same system.” Traditional software chains are too heavy for these small but meaningful requests, while pure chatbot interfaces often fail to guarantee scientific rigor. QCViz-MCP v3 aims to occupy the practical middle ground between those two extremes by connecting a real computation engine with a real user-facing workflow.

## 3. 시스템이 실제로 하는 일 / What the System Actually Does

사용자는 웹 브라우저에서 한국어 또는 영어로 분자와 계산 의도를 입력할 수 있다. 입력은 단순한 정식 분자명뿐 아니라, 띄어쓰기 잡음이 포함된 한국어 표기, 혼합 언어 입력, 약어, 설명형 질의, 후속 질문 형태로 들어올 수 있다. 시스템은 우선 이 입력을 정규화하고, 구조에 대한 직접 질의인지, 설명이 필요한 대화형 질의인지, 또는 구조 grounding이 먼저 필요한 반정형 질의인지를 분류한다. 그런 다음 MolChat 또는 내부 정규화/해석 계층을 통해 분자 후보를 찾고, 필요한 경우 clarification form을 생성하여 사용자가 의미하는 실제 분자를 확인한다.

The user can enter a molecule and an analysis intent in Korean or English from the browser. The input may be a clean canonical molecule name, a noisy Korean string with spacing artifacts, a mixed-language expression, an acronym, a descriptive phrase, or a follow-up question that depends on previous context. The system first normalizes this input and classifies whether it is a direct structure request, a conversational question that should stay in chat mode, or a semi-structured request that requires semantic grounding before computation. It then uses MolChat and internal normalization layers to identify candidate molecules and, when needed, generates a clarification form so that the user can confirm the intended structure.

구조가 계산 가능한 상태로 잠기면, 시스템은 해당 요청에 맞는 계산 경로를 선택한다. 오비탈 프리뷰, ESP 맵, 부분 전하, 구조 최적화, 단일점 계산, 분석용 요약 등은 서로 다른 후처리와 결과 계약을 필요로 한다. 따라서 이 플랫폼은 단순히 하나의 계산 함수를 호출하는 것이 아니라, 사용자의 의도, 구조 상태, 세션 맥락, 계산 파라미터, 선택된 시각화 모드에 따라 서로 다른 실행 및 결과 패키징 경로를 사용한다. 계산이 끝난 뒤에는 결과를 단순 숫자나 파일 경로로 끝내지 않고, 브라우저에서 즉시 읽을 수 있는 설명, 시각화 데이터, 후속 질문에 재사용 가능한 상태 정보와 함께 반환한다.

Once the structure is locked into a computation-ready state, the system chooses an execution path that matches the request. Orbital preview, ESP mapping, partial charges, geometry optimization, single-point evaluation, and explanatory summaries each require a different post-processing contract. For that reason, the platform does not merely call one generic calculation function. Instead, it uses distinct execution and packaging paths depending on the user intent, structure status, session context, calculation parameters, and visualization mode. After the calculation, the result is not left as a raw number or a file path. It is returned together with browser-readable explanation, visualization data, and reusable state for follow-up questions.

## 4. 아키텍처 개요 / Architectural Overview

프로젝트의 상위 구조는 크게 웹 계층, 대화/계획 계층, 구조 해석 및 외부 연동 계층, 계산 계층, 후처리 및 시각화 계층으로 구분할 수 있다. 웹 계층은 FastAPI를 중심으로 REST와 WebSocket 인터페이스를 제공하며, 세션, 인증, 관리자 기능, 상태 조회를 담당한다. 대화/계획 계층은 사용자 입력을 정규화하고, 현재 입력이 chat-only인지 grounding-required인지 compute-ready인지 판단하며, clarification이 필요한 경우 적절한 폼과 후보 목록을 만들어낸다. 구조 해석 계층은 MolChat, PubChem, 한국어 alias 정규화, 이온쌍 처리 로직을 조합하여 구조 후보를 계산 가능한 형태로 정리한다. 계산 계층은 PySCF 기반 실행 경로와 작업 큐를 담당하며, 후처리 계층은 시각화 데이터, 설명, advisor 결과를 만들어 브라우저에 전달한다.

The top-level structure of the project can be described in terms of a web layer, a conversational planning layer, a structure grounding and external integration layer, a computation layer, and a post-processing and visualization layer. The web layer is built around FastAPI and exposes REST and WebSocket interfaces while also handling sessions, authentication, administration, and status inspection. The conversational planning layer normalizes user input, decides whether the input belongs to chat-only, grounding-required, or compute-ready routing, and generates clarification forms and candidate lists when needed. The structure grounding layer combines MolChat, PubChem, Korean alias normalization, and ion-pair handling to transform user phrases into computation-ready structures. The computation layer manages PySCF-backed execution paths and queueing, while the post-processing layer builds visualization payloads, explanations, and advisor outputs for browser delivery.

이 구조에서 중요한 점은 각 계층이 명확한 역할을 가지고 있다는 것이다. QCViz-MCP v3는 하나의 거대한 “에이전트”가 모든 것을 처리하는 방식이 아니라, 불확실성이 높은 자연어 단계와 엄밀성이 요구되는 계산 단계를 분리하는 방식을 채택했다. 그 결과 시스템은 구조 해석 실패, clarification 필요, 계산 실패, 후속 질문 재사용 등 서로 다른 상태를 더 명시적으로 관리할 수 있으며, 대화형 인터페이스와 계산 엔진 사이의 계약을 테스트하기도 쉬워진다.

What matters in this architecture is that each layer has a distinct role. QCViz-MCP v3 is not built around one monolithic “agent” that attempts to do everything. Instead, it explicitly separates the high-uncertainty language stage from the high-rigor computation stage. As a result, the system can manage different states more explicitly, including structure interpretation failure, clarification-needed transitions, computation failure, and state reuse across follow-up turns. It also becomes easier to test the contract between the conversational interface and the scientific backend.

## 5. LLM, MolChat, MCP, PySCF의 역할 분담 / Functional Roles of LLM, MolChat, MCP, and PySCF

이 프로젝트에서 LLM은 계산값을 생성하는 주체가 아니다. LLM의 주 역할은 사용자의 언어적 표현을 작업 가능한 의도와 구조 후보로 정리하는 것이다. 예를 들어 약어가 포함된 질문, 설명형 질의, 구조와 작업 요청이 혼합된 문장을 만났을 때, 시스템은 대화형 답변을 해야 하는지, 구조 grounding을 해야 하는지, 바로 계산으로 들어가도 되는지를 먼저 판단해야 한다. 이 과정에서 LLM 또는 LLM 스타일의 계획 계층은 자연어를 계산 워크플로우로 연결하는 해석자 역할을 한다. 그러나 그 자체가 계산 결과의 근거가 되지는 않는다.

Within this project, the LLM is not the entity that produces scientific values. Its primary role is to reorganize a user’s linguistic expression into a workable intent and a set of structure candidates. When the system encounters an acronym-heavy query, a descriptive phrase, or a sentence that mixes structure reference and task request, it must first decide whether to stay in conversational mode, perform semantic grounding, or proceed directly to computation. In that sense, the LLM or LLM-style planning layer acts as an interpreter that connects natural language to a computation workflow. It is not, by itself, the scientific source of the result.

MolChat는 이 프로젝트에서 매우 중요한 grounding 파트너다. QCViz-MCP v3가 단순한 이름-조회 시스템을 넘어서는 이유 중 하나는, 사용자가 항상 정식 분자명이나 완전한 SMILES를 제공하지 않는다는 사실을 전제로 설계되었기 때문이다. “TNT에 들어가는 주물질” 같은 설명형 입력은 MolChat의 해석 경로를 통해 분자 후보로 바뀌고, QCViz는 이 후보를 clarification dropdown으로 전달하여 사용자가 의도한 구조를 선택할 수 있게 만든다. 즉 MolChat는 단순 보조 API가 아니라, 챗봇형 자연어 질의를 실제 계산용 구조 후보로 바꾸는 semantic grounding 계층으로 동작한다.

MolChat is a highly important grounding partner in this project. One reason QCViz-MCP v3 goes beyond a simple name-resolution interface is that it assumes users will not always provide a clean canonical name or a fully specified SMILES string. A descriptive query such as “the main component used in TNT” is turned into candidate molecules through MolChat’s interpretation path, and QCViz forwards those candidates into a clarification dropdown so that the user can select the intended structure. In other words, MolChat is not a minor helper API. It functions as the semantic grounding layer that maps chatbot-style natural language into actual structure candidates for computation.

MCP는 보다 넓은 의미에서 LLM과 외부 계산 도구를 연결하는 개념적 프레임을 제공한다. 프로젝트 내부 구현은 웹 앱 중심 구조로 진화해 왔지만, 기본 철학은 동일하다. 언어 모델이 생각을 담당하고, 외부 도구가 행동을 담당하며, 그 사이에는 검증 가능한 구조화된 계약이 있어야 한다는 점이다. 이 철학 덕분에 시스템은 “그럴듯한 답변”과 “실제 계산 결과”를 분리해서 다룰 수 있다.

MCP, in a broader sense, provides the conceptual frame for connecting language models to external computational tools. Although the project implementation has evolved toward a web-first architecture, the underlying philosophy remains the same: the language model handles interpretation, the external tool handles action, and the interface between them must be governed by a structured, testable contract. This philosophy is what allows the system to separate plausible language output from actual computational results.

PySCF는 이 시스템의 과학적 엔진이다. 오비탈, ESP, 부분 전하, 구조 최적화 등 실제 계산 결과는 PySCF 기반 실행 경로를 통해 생성된다. 따라서 이 플랫폼의 신뢰성은 결국 PySCF 경로와 그 후처리 품질에 달려 있다. QCViz-MCP v3의 가치는 “AI가 계산을 대신한다”가 아니라 “PySCF와 같은 실제 계산 엔진에 실험자가 쉽게 도달하게 한다”는 데 있다.

PySCF is the scientific engine of the system. Orbitals, electrostatic potential maps, partial charges, and geometry optimization outputs are produced through a PySCF-based execution path. The scientific credibility of the platform therefore depends on the PySCF route and the quality of the downstream post-processing. The value of QCViz-MCP v3 is not that “AI replaces quantum chemistry,” but that it allows experimentalists to reach a real engine such as PySCF more easily and more interactively.

## 6. 사용자 경험과 인터랙션 설계 / User Experience and Interaction Design

이 프로젝트의 강점은 사용자가 단순히 계산을 던지고 결과 파일을 다운로드받는 구조가 아니라는 점에 있다. 시스템은 사용자의 입력이 모호할 때 적절한 clarification form을 생성하고, 이전 세션에서 계산한 구조를 후속 질문에 재사용하며, 현재 입력이 설명 요청인지 계산 요청인지 최대한 안정적으로 구분하려고 한다. “MEA알아?” 같은 질문은 설명형 대화로 다루고, “MEA HOMO 보여줘” 같은 입력은 약어 기반 구조 grounding 후 계산 경로로 보내도록 라우팅을 세분화하는 식이다. 이러한 구분은 단지 UI 편의 기능이 아니라, 실제 scientific workflow의 오류를 줄이기 위한 제어 장치다.

One of the strengths of the project is that the user is not merely sending a calculation and receiving a downloadable file. When the input is ambiguous, the system generates a clarification form. When a follow-up question depends on a previous session structure, it attempts to reuse that context. When a new input resembles a conceptual question rather than a computation request, the system tries to keep it in chat mode. A question such as “Do you know MEA?” is treated as an explanatory conversation, whereas a request like “Show me the HOMO of MEA” is routed through acronym grounding and then into a computational path. This distinction is not just a UI convenience. It is a control mechanism for reducing real scientific workflow errors.

또한 이 시스템은 clarification을 누적 카드 형태로 방치하지 않고, 현재 턴에 해당하는 확인 과정만 유효하게 유지하려고 설계되었다. 이전 턴의 결과가 새 질문에 잘못 붙는 문제, 이전 세션의 메시지가 현재 세션 대화에 섞이는 문제, canonical molecule 선택 후 다시 composition heuristic이 발동하는 문제 등은 실제 사용자 신뢰를 크게 떨어뜨리는 요소이기 때문에, 채팅 상태 무결성과 결과-턴 바인딩은 이 프로젝트에서 매우 중요한 품질 영역으로 다뤄졌다.

In addition, the system is designed to avoid leaving clarification cards as a messy append-only sequence. It tries to keep only the currently relevant confirmation state active for the current turn. Problems such as attaching an old result to a new question, mixing previous-session messages into the present conversation, or re-triggering composition heuristics after a canonical molecule has already been selected are severe trust failures from the user’s perspective. For that reason, chat-state integrity and result-to-turn binding have been treated as critical quality dimensions in this project.

## 7. 복잡한 시각화 툴체인 없이 끝까지 가는 플랫폼 / A Platform That Completes the Workflow Without a Complex Visualization Toolchain

이 프로젝트의 가장 실질적인 강점 중 하나는 계산을 수행하는 데서 멈추지 않고, 결과를 별도의 전문 시각화 프로그램 없이도 브라우저에서 바로 확인할 수 있게 만든다는 점이다. 전통적인 계산 워크플로에서는 계산과 시각화가 서로 다른 소프트웨어 체인으로 분리되는 경우가 많다. 사용자는 출력 파일을 정리하고, cube나 표면 데이터를 생성하고, 별도 뷰어를 열어 렌더링하고, 다시 이미지를 캡처하거나 수치를 읽어야 한다. QCViz-MCP v3는 이 공정을 브라우저 안으로 최대한 끌어와서, 구조, HOMO/LUMO, ESP, 요약 설명을 한 플랫폼 안에서 연결한다.

One of the most practical strengths of the project is that it does not stop at calculation execution. It delivers the result in a form that can be inspected directly in the browser without requiring a separate specialist visualization package. In traditional computational workflows, calculation and visualization often live in separate software chains. The user has to prepare output files, generate cube or surface data, open a dedicated viewer, render the data, and then capture images or read out values. QCViz-MCP v3 pulls as much of this process as possible into the browser so that structure, HOMO/LUMO, ESP, and explanatory summary can be accessed within one platform.

이 점은 단순히 화면이 보기 좋다는 의미가 아니다. 그것은 계산 완료 이후의 post-processing friction을 줄이고, 파일 포맷 hopping을 줄이며, 결과를 실제 토론 가능한 단위로 재포장한다는 의미다. 특히 실험 연구자에게 중요한 것은 “계산이 끝났다”는 사실보다 “내가 이 결과를 바로 보고 해석할 수 있느냐”이다. 이 프로젝트는 그 간극을 줄인다. 따라서 이 시스템의 contribution을 설명할 때에는 “계산 엔진 호출”만 강조하는 것이 아니라, “계산에서 시각화까지의 마지막 구간을 플랫폼 안에서 닫아준다”는 점을 명시적으로 강조해야 한다.

This is not merely a matter of visual polish. It means reducing post-processing friction after the calculation has completed, reducing file-format hopping, and repackaging the output into a unit that can actually support interpretation and discussion. For experimental researchers, what matters is often not simply that “the computation has finished,” but whether the result is immediately visible and interpretable. This project narrows that gap. Accordingly, when the contribution is presented, the emphasis should not stop at “calling a quantum chemistry engine,” but should explicitly include the claim that the platform closes the last mile from calculation to visualization.

## 8. 핵심 컨트리뷰션 정리 / Core Contributions

이 프로젝트의 첫 번째 기여는 실험자 관점의 브라우저 기반 end-to-end 워크플로우를 실제 코드와 UI 상태 기계 수준에서 구현했다는 점이다. 사용자는 별도의 로컬 계산 체인이나 전문 시각화 도구를 직접 다루지 않고도, 질문에서 구조 확인, 계산 실행, 결과 확인, 후속 질문까지 이어지는 흐름을 하나의 브라우저 인터페이스 안에서 경험할 수 있다. 이 contribution은 단순 “web front-end 제공”이 아니라, 계산과 해석 사이에 놓인 실제 사용성 장벽을 줄이는 시스템 설계라는 점에서 의미가 있다.

The first contribution of the project is the implementation of an experimentalist-facing, browser-based, end-to-end workflow at the level of actual code and UI state management. The user can move from question to structure confirmation, calculation execution, result inspection, and follow-up interaction without directly managing a local computation chain or a separate specialist visualization tool. This contribution is more than simply providing a web front-end. It is a system design effort aimed at reducing the real usability barrier between computation and interpretation.

두 번째 기여는 messy real-world chemistry query를 계산 가능한 구조로 연결하기 위한 semantic grounding 계층을 강화했다는 점이다. 이 플랫폼은 정식 분자명뿐 아니라 한국어 입력, 띄어쓰기 오류, 설명형 질의, 약어 기반 요청, 이온쌍 또는 혼합 입력까지 다루며, 필요할 경우 MolChat를 grounding 파트너로 사용해 후보를 구조화된 dropdown으로 제시한다. 이는 사용자의 실제 입력 습관을 계산 가능한 구조 표현으로 번역하는 작업이며, 단순 사전 조회 이상의 의미를 가진다.

The second contribution is the strengthening of a semantic grounding layer that links messy real-world chemistry queries to computation-ready structures. The platform handles not only canonical names but also Korean input, spacing errors, descriptive phrases, acronym-driven requests, and ion-pair or mixed inputs. When necessary, it uses MolChat as a grounding partner and presents candidate molecules through a structured clarification dropdown. This is not a simple dictionary lookup. It is a system for translating real user language habits into computation-ready molecular representations.

세 번째 기여는 대화형 continuation과 clarification을 scientific workflow control로 재해석했다는 점이다. QCViz-MCP v3는 후속 질문에서 이전 구조를 재사용할 수 있게 만들되, 세션이 다르거나 구조 잠금이 없는 경우에는 continuation targeting clarification으로 멈추게 하며, 질문형 대화와 계산형 요청을 분리하는 routing 규칙을 정교화했다. 이는 대화 인터페이스의 편의성을 높이는 동시에, 잘못된 구조로 계산이 이어지는 위험을 줄이는 안전장치다.

The third contribution lies in reinterpreting conversational continuation and clarification as scientific workflow control rather than mere chat convenience. QCViz-MCP v3 supports follow-up requests that reuse a previous structure, but when the relevant session state is missing it stops the process through a continuation-targeting clarification. It also refines routing rules that separate explanatory conversation from actual computational intent. This increases the convenience of the conversational interface while simultaneously reducing the risk of silently continuing computation on an unintended structure.

네 번째 기여는 계산 결과를 파일이나 숫자 수준에서 끝내지 않고, 설명과 시각화를 포함한 해석 가능한 산출물로 묶는다는 점이다. 오비탈, ESP, 구조 정보, 전하 분석, advisor 계층의 요약은 각각 독립적인 파일 조각으로 흩어지지 않고 브라우저 중심 결과 인터페이스 안에 결합된다. 이 contribution은 실험자와 비전문 사용자에게 특히 중요하며, 기존의 다중 프로그램 시각화 체인을 단축한다.

The fourth contribution is that the project does not stop at files or raw numeric output. It packages the result as an interpretable outcome that includes explanation and visualization. Orbitals, ESP, structural information, charge analysis, and advisor-layer summaries are not left as disconnected file fragments. They are integrated into a browser-centered result interface. This contribution is especially important for experimentalists and non-specialist users because it shortens the traditional multi-program visualization chain.

## 9. 노벨티와 포지셔닝에 대한 판단 / Judgment on Novelty and Positioning

현 시점에서 “LLM + MCP + PySCF를 연결했다”는 문장만으로는 강한 노벨티를 주장하기 어렵다. 이 조합은 이미 다양한 proof-of-concept와 agentic chemistry 프로젝트 안에서 등장했고, 리뷰어 역시 같은 키워드의 반복에 민감할 가능성이 높다. 따라서 이 프로젝트를 논문화할 때에는 조합 자체를 메인 메시지로 내세우기보다, 실험자 대상 웹 기반 플랫폼, semantic grounding, clarification-driven workflow control, external visualization toolchain reduction, browser-native result experience라는 시스템 차원의 contribution을 중심에 두는 것이 타당하다.

At the present moment, it is difficult to claim strong novelty on the basis of the sentence “we connected LLM, MCP, and PySCF.” That combination has already appeared across multiple proof-of-concept and agentic chemistry efforts, and reviewers are likely to be sensitive to repeated keyword-level novelty claims. For publication purposes, the project should therefore not center its claim on the tool combination itself. Instead, it should foreground system-level contributions such as an experimentalist-facing web platform, semantic grounding, clarification-driven workflow control, reduction of dependence on external visualization toolchains, and a browser-native result experience.

이 판단은 프로젝트의 가치를 깎아내리는 것이 아니다. 오히려 contribution의 단위를 더 정확하게 잡자는 제안에 가깝다. 본 프로젝트는 새로운 전자구조 이론을 제안하는 논문보다는, 계산 접근성을 바꾸는 플랫폼 논문, 또는 사용자 워크플로우를 재구성하는 software/system paper로서 더 강한 위치를 가진다. 이 관점에서 보면 Journal of Cheminformatics, Digital Discovery, JCIM 등은 충분히 현실적인 타깃이 될 수 있다.

This judgment is not a dismissal of the project’s value. It is a proposal to define the unit of contribution more accurately. The project is not strongest as a paper claiming a new electronic-structure theory. It is stronger as a platform paper that changes computational accessibility, or as a software/system paper that restructures the user workflow. From that perspective, journals such as Journal of Cheminformatics, Digital Discovery, and JCIM are realistic and defensible targets.

## 10. 향후 확장과 제출 전략 / Future Extensions and Submission Strategy

프로젝트의 다음 단계는 두 갈래로 볼 수 있다. 하나는 시스템 자체를 더 안정화하고 확장하는 경로이다. 여기에는 보다 풍부한 semantic grounding benchmark, 더 다양한 계산 기능, 더 강한 오류 회복, 멀티유저 운영 안정성, 실사용 세션 로깅과 관리자 도구 강화가 포함된다. 다른 하나는 논문을 위한 증거 체계를 만드는 경로이다. 여기에는 실험 연구자 대상 user study, 기존 워크플로 대비 시간 절감 비교, 오류율 또는 task completion rate 측정, multilingual/noisy input benchmark, 구조 grounding 성공률 측정이 포함될 수 있다. 이 두 갈래가 결합될 때, 프로젝트는 단순한 데모가 아니라 검증 가능한 플랫폼 연구로 자리 잡을 수 있다.

The next phase of the project can be viewed along two tracks. One track is to stabilize and extend the system itself. This includes stronger semantic grounding benchmarks, broader computational capabilities, better recovery from error conditions, improved multi-user operational stability, and stronger session logging and administration tools. The other track is to build the evidentiary basis required for publication. This may include a user study with experimental researchers, comparisons of task completion time against traditional workflows, error-rate or completion-rate measurements, multilingual and noisy-input benchmarks, and success-rate evaluation for structure grounding. When these two tracks are developed together, the project can be presented not merely as a demonstration but as a validated platform study.

제출 전략 측면에서는 contribution을 과장하는 것보다 명확하게 제한하는 편이 유리하다. 이 시스템은 범용 과학 에이전트 전체를 대체한다고 주장할 필요가 없다. 대신 실험자 관점에서 가장 반복적으로 발생하는 소규모 양자화학 질의와 시각화 요청을 브라우저 기반 대화형 시스템으로 흡수했다는 점을 분명히 하는 편이 더 설득력 있다. 특히 “계산 실행”만이 아니라 “계산에서 시각화와 해석 가능한 결과까지의 마지막 구간을 닫아주는 플랫폼”이라는 메시지는 이 프로젝트의 실질적 강점을 잘 전달한다.

In terms of submission strategy, it is better to define the claims clearly than to overstate them. The system does not need to claim that it replaces the entire universe of general-purpose scientific agents. It is more persuasive to state that the platform absorbs a recurring class of small but meaningful quantum chemistry and visualization requests into a browser-based conversational system for experimentalists. In particular, the message that the platform closes not only the execution step but also the last mile from calculation to interpretable visualization is a strong and accurate summary of the project’s practical value.

## 11. 결론 / Conclusion

QCViz-MCP v3는 자연어 기반 계산 자동화, semantic grounding, 대화형 clarification, PySCF 실행, 브라우저 시각화, 후속 질문 상태 재사용을 하나로 묶은 실험자 지향형 플랫폼이다. 이 프로젝트의 강점은 단순히 새로운 도구 조합을 시도했다는 데 있지 않다. 오히려 기존 계산 워크플로우에서 가장 번거롭고 분절된 구간들, 즉 구조 해석, 계산 진입, 후속 확인, 시각화, 결과 해석을 하나의 일관된 사용자 경험으로 묶는 데 있다. 따라서 이 시스템은 “LLM이 양자화학을 한다”는 식의 과장된 서사보다는, “실제 계산 엔진과 실제 사용자 워크플로우를 연결하는 대화형 플랫폼”으로 설명될 때 가장 정확하고 가장 강하다.

QCViz-MCP v3 is an experimentalist-oriented platform that unifies natural-language-driven computation, semantic grounding, conversational clarification, PySCF execution, browser-native visualization, and reusable session context for follow-up questions. Its strength does not primarily lie in trying a novel keyword combination. Rather, it lies in binding together the most fragmented and inconvenient parts of the existing computational workflow: structure interpretation, computational entry, confirmation, visualization, and result interpretation. For that reason, the project is strongest not when it is framed as “the LLM does quantum chemistry,” but when it is presented as a conversational platform that connects real computation engines to real scientific user workflows.
